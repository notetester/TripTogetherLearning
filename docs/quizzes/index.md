# 퀴즈 모음 — 면접 전 셀프체크

> 읽고 끄덕이는 것과 면접에서 말로 설명하는 것은 다른 능력입니다. 이 코너는 TripTogether를 **설명할 수 있는지** 스스로 검증하는 곳입니다.

## 1. 이 코너는 무엇인가

TripTogether 학습 문서의 각 도메인 페이지에는 끝에 `## 퀴즈` 섹션이 있고, 객관식 위젯(`QuizBox`)으로 핵심 개념을 확인합니다. 이 페이지는 그 퀴즈들을 **영역별로 묶은 허브**이자, 면접 직전에 빠르게 돌릴 수 있는 **종합 셀프체크 시작점**입니다.

영역별 퀴즈는 4개로 나뉩니다.

| 묶음 | 다루는 범위 | 들어가기 |
| --- | --- | --- |
| **백엔드 퀴즈** | Spring Boot·MyBatis·JSP·인터셉터·AOP·세션·소프트삭제 | [백엔드 퀴즈 →](/quizzes/backend) |
| **도메인 퀴즈** | 인증·커뮤니티·문의·코스·탐색·커머스·관리자·i18n | [도메인 퀴즈 →](/quizzes/domains) |
| **AI 퀴즈** | 다중 모델·구조화 출력·폴백·쿼터·모더레이션 | [AI 퀴즈 →](/quizzes/ai) |
| **프로젝트 Q&A** | 설계 의도·트레이드오프·구현 상태·전체 흐름 | [프로젝트 Q&A →](/quizzes/project-qna) |

## 2. 왜 영역별로 나눴나

면접은 보통 **한 영역을 깊게** 파고듭니다. 백엔드 면접관은 MyBatis와 트랜잭션을, AI를 보는 면접관은 모델 선택 근거와 폴백을 묻습니다. 한 덩어리로 섞어 두면 약한 영역을 골라 집중 연습하기 어렵습니다. 그래서 백엔드·도메인·AI·종합으로 분리해, **자기 약점만 반복**할 수 있게 했습니다.

또한 TripTogether는 4명이 도메인을 나눠 만든 팀 프로젝트입니다. 자기가 직접 짠 영역은 빠르게 통과하고, **상대적으로 덜 만진 영역**(예: 다른 사람이 담당한 도메인)을 골라 보완하는 용도로도 쓸 수 있습니다.

## 3. 어떻게 쓰나

각 퀴즈 페이지에서 보기를 고르면 정답·해설이 즉시 표시됩니다. 권장 순서는 다음과 같습니다.

1. **틀 잡기** — [학습/말하기 전략](/guide/how-to-study)으로 설명 프레임을 먼저 익힙니다.
2. **약한 영역 진단** — 위 4개 묶음을 한 바퀴 돌려 어디서 막히는지 확인합니다.
3. **해당 도메인 정독** — 막힌 영역의 도메인 페이지([인증](/auth/)·[커뮤니티](/community/)·[코스](/courses/)·[탐색·커머스](/explore/) 등) 본문을 다시 읽습니다.
4. **재시도 + 입으로 말하기** — 같은 퀴즈를 다시 풀고, 정답을 **소리 내어 근거까지** 설명합니다.

:::tip 정답을 맞히는 게 목적이 아니다
객관식을 맞혔다고 끝이 아닙니다. 각 문제의 `explanation`에 적힌 근거를 **자기 문장으로 다시 말할 수 있어야** 통과입니다. 보기를 외우지 말고, 왜 그 보기가 정답인지를 설명하세요.
:::

## 4. 출제 근거

모든 퀴즈는 추측이 아니라 **실제 코드·DB·문서**에 근거합니다.

- **클래스 단위** — `AiPlanServiceImpl`, `AssistantServiceImpl`, `ChatbotService`, `AuthorizationAspect`, `LoginUserArgumentResolver` 등 실존 클래스의 동작을 묻습니다.
- **테이블 단위** — `TRAVEL_PLAN`·`PLAN_SPOT`, `COMMUNITY_TAG`·`POST_TAG`·`TAG_RELATION`, `ADMIN_ACTION_AUDIT`, `USER_WALLET_HISTORY` 등 스키마 기반입니다.
- **설계 결정** — ADR(MADR) 0001~0014의 의도(소프트삭제·XSS 정화·AI 모더레이션·CSRF 부분 적용 등)를 묻습니다.
- **구현 상태** — 실제로 동작하는 기능과 Mock·계획 단계를 정직하게 구분해 출제합니다.

## 5. 구현 상태 주의 — 헷갈리기 쉬운 경계

퀴즈에는 다음처럼 **됨 vs Mock/계획**을 구분하는 함정 문제가 섞여 있습니다. 미리 정리해 두면 안 틀립니다.

| 항목 | 상태 |
| --- | --- |
| GPT 멀티턴 어시스턴트 / AI 일정 생성 | 실제 외부 API 연동 (구현됨) |
| Gemini 네비 챗봇 / 여행지 추천 | 실제 외부 API 연동 (구현됨) |
| Claude 문의 답변 초안 / Perspective 독성 감지 | 실제 외부 API 연동 (구현됨) |
| 항공권 검색 (`FlightOfferProvider`) | 인터페이스만 추상화, 실제 항공 API 미연동 (Mock) |
| AI 응답 품질 정량 평가 | 체계 부재, 향후 과제 (계획) |
| 모바일 레이아웃 | JSP 데스크톱 위주, 반응형·SPA는 향후 (계획) |

:::warning 단골 함정
"실제 외부 API에 연동되지 않은 기능은?" 같은 문제의 정답은 거의 항상 **항공권(Mock)**입니다. AI 4종은 모두 실제 연동, 항공권만 Mock이라는 점을 분리해서 기억하세요.
:::
