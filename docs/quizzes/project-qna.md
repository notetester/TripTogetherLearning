# 프로젝트 Q&A 점검 — 주관식 면접 시뮬레이션

> 객관식으로는 "아는지"만 확인된다. 면접에서 진짜로 평가받는 건 **막힘 없이 30초·1분·3분으로 말하는 능력**이다. 이 페이지는 TripTogether를 입으로 설명하는 주관식 리허설이다. 답을 외우지 말고, 빈칸을 자기 언어로 채워 소리 내어 말해 본다.

TripTogether는 4명이 14~15개 도메인을 수직 분담해 만든 팀 프로젝트다. 본인이 맡은 도메인은 일반 화법으로 자연스럽게 설명하되("저는 인증 모듈을 맡아…"), 나머지 도메인도 동등하게 사실 기반으로 말할 수 있어야 한다. 특정 도메인만 과대평가하거나, 남의 도메인을 깎아내리지 않는다.

허브: [도메인 전체 개요](/domains) · [담당별 보기](/by-area/) · [전체 흐름](/flow/) · [프로젝트 전체 플레이북](/flow/interview-whole-project)

## 1. 한 줄 정의

이 페이지는 정답 텍스트가 아니라 **말하기 훈련 대본**이다. 각 질문마다 (1) 핵심 한 줄, (2) 자기 언어로 펼칠 뼈대, (3) 흔한 함정을 준다. 녹음하면서 시간을 재고, 정해진 초 안에 끝나는지 확인한다.

## 2. 왜 주관식 리허설이 필요한가

- **객관식은 재인(recognition), 면접은 인출(recall)이다.** 보기를 고르는 것과 백지에서 말하는 것은 난이도가 다르다.
- **첫 30초가 인상을 결정한다.** "무슨 서비스인가"를 한 호흡에 말하지 못하면 그 뒤가 아무리 좋아도 손해다.
- **꼬리질문은 깊이를 캔다.** "왜 그렇게 했나"에 근거(클래스·테이블·ADR)로 답해야 표면 지식과 구분된다.
- **정직함이 가산점이다.** 항공권 Mock, AI 품질 평가 부재 같은 한계를 먼저 말하고 개선안을 붙이면 신뢰가 올라간다.

:::tip 리허설 규칙
타이머를 켜고, 끊기면 멈추지 말고 끝까지 간 뒤 다시 한다. 3번 반복하면 문장이 손에 붙는다.
:::

## 4. 동작 원리 — 길이별 대본

### 30초 — 엘리베이터 피치

> TripTogether는 국내 여행을 탐색부터 계획, 예약, 공유까지 한 곳에서 처리하는 올인원 플랫폼입니다. Spring Boot 단일 애플리케이션에 도메인을 수직 분담하고, 인증·로깅·알림·국제화 같은 횡단 관심사는 인터셉터와 AOP로 공통화했습니다. 도메인별로 GPT, Gemini, Claude, Perspective를 작업 성격에 맞게 붙여 지능형 기능을 제공합니다.

- 함정: 스택만 나열하고 "무슨 서비스인지"를 빼면 안 된다. 첫 문장은 반드시 사용자 가치다.

### 1분 — 구조 한 겹 추가

30초 피치에 한 요청의 일생을 덧붙인다.

1. 한 요청은 인터셉터 체인(locale → ipBlock → activityLog → login → admin → notification)을 지나
2. AOP 권한 검사(`@RequireLogin`/`@RequireAdmin`)를 받고
3. service → MyBatis mapper → MySQL로 내려갔다가
4. JSP로 렌더되거나 SSE 알림으로 사용자에게 돌아온다.

- 함정: 4계층을 외운 티만 내지 말고, "그래서 4명이 일관되게 만들 수 있었다"는 결론으로 닫는다.

### 3분 — 도메인 + 한계까지

1분 대본 뒤에 (1) 본인 담당 도메인을 깊게 한 사례로, (2) AI 다중 모델 전략, (3) 구현 한계(항공권 Mock·AI 품질 평가 부재·모바일·Swagger)와 개선안 순으로 확장한다.

## 5. 핵심 주관식 5문 — 뼈대와 함정

### Q1. 프로젝트를 설명해 보세요

- 핵심 한 줄: 탐색 → 계획 → 예약 → 공유 올인원, 모놀리식 수직 분담, 다중 AI.
- 뼈대: 사용자 가치 → 스택 한 줄 → 공통 골격 → AI 위임.
- 함정: 기능을 20개 나열하는 것. 큰 그림 → 한 사례로 좁혀라.

### Q2. 본인은 어떤 역할이었나요

- 핵심 한 줄: "저는 [담당 도메인] 모듈을 컨트롤러부터 매퍼·화면까지 수직으로 맡았습니다."
- 뼈대: 담당 범위 → 대표 클래스/테이블 한두 개 → 직접 해결한 문제 1개 → 협업 시 공통 규칙 준수.
- 함정: "팀이 했다"로 뭉개거나, 반대로 남의 도메인까지 자기 공으로 말하기. 본인 기여는 일반 화법으로 분명히, 남의 도메인은 사실대로.

:::warning 역할 화법
"내 영역/내가 다 했다"식 과장도, "잘 모른다"식 회피도 감점이다. 담당은 구체적 산출물로, 협업은 공통 규칙(4계층·소프트 삭제·ApiResponse 같은 일관성)으로 설명한다.
:::

### Q3. 가장 어려웠던 점은

- 핵심 한 줄: 한 가지 구체적 문제 + 해결 과정 + 배운 점.
- 뼈대 예시(택1):
  - 다중 행 정합성 — AI 일정 생성에서 TRAVEL_PLAN과 plan_spot을 함께 저장하다 중간 실패가 났고, @Transactional로 묶어 전부 롤백되게 했다.
  - 실시간 알림 — SseEmitter가 다중 탭·타임아웃에서 끊겨, userIdx를 List로 관리하고 하트비트와 DB 저장 후 푸시 안전망을 넣었다.
  - 외부 AI 불안정 — Gemini 추천이 실패하면 DB 캐시 → 트렌딩 순으로 폴백해 사용자에게는 항상 결과가 나가게 했다.
- 함정: "협업이 힘들었다" 같은 추상적 답. 기술적·구체적 문제 하나를 끝까지 파라.

### Q4. 기술 선택의 이유

- 핵심 한 줄: 각 선택에 "그 상황에서 더 단순했기 때문"이라는 이유를 붙인다.
- 뼈대:
  - MyBatis(JPA 아님) — 탐색 7탭 동적 필터와 통계 집계처럼 SQL 직접 제어가 많아서.
  - JSP 서버 렌더링 — 세션 인증 + i18n 메시지 결합이 단순하고 빨라서(SPA·반응형은 향후 과제로 명시).
  - 다중 AI — 멀티턴·구조화 출력은 GPT-4o-mini, 네비·추천은 Gemini 2.5 Flash, 문의 초안은 Claude Haiku, 독성은 Perspective. 모델은 다양화하되 호출 골격은 통일.
- 함정: "최신이라서/유행이라서"는 금물. 트레이드오프를 인정하라.

### Q5. 다시 한다면 무엇을 개선할까

- 핵심 한 줄: 한계를 먼저 정직하게, 그다음 구체적 개선안.
- 뼈대: AI 호출에 품질·지연·실패율을 측정하는 관측 레이어를 처음부터 / 화면을 API + SPA로 분리해 모바일·협업 개선 / 항공권은 `FlightOfferProvider` 인터페이스가 이미 있으니 실제 API 어댑터만 교체.
- 함정: "딱히 없다"는 최악의 답. 한계 인지 자체가 역량 신호다.

## 6. 면접 답변 3단계 (자기 역할 버전)

본인 역할을 물었을 때 쓰는 3단계 템플릿이다.

1. **범위** — "저는 [담당 도메인]을 컨트롤러부터 매퍼, 화면까지 수직으로 담당했습니다."
2. **근거** — "대표적으로 [클래스/테이블]을 설계했고, [구체적 문제]를 [해결책]으로 풀었습니다."
3. **협업** — "팀 전체는 4계층·소프트 삭제·세션 인증 같은 공통 규칙으로 일관성을 맞췄고, 공통 영역 변경은 합의 후 진행했습니다."

## 7. 꼬리질문 + 모범답안

:::details "그 도메인에서 본인이 직접 짠 코드는 무엇인가요?"
담당 도메인의 ServiceImpl 한 곳을 골라, 메서드 하나의 입력·처리·출력과 예외 경로를 말합니다. 예를 들어 다중 행을 쓰는 메서드라면 @Transactional 경계와 실패 시 롤백 동작까지 설명합니다. 추상적으로 "전반을 했다"가 아니라 클래스·메서드 이름을 댈 수 있어야 합니다.
:::

:::details "팀 프로젝트에서 본인 기여를 어떻게 증명하나요?"
커밋 단위 작업 분담과 PR 흐름으로 증명합니다. 개인 브랜치에서 작업해 dev로 PR했고, 담당 폴더(수직 슬라이스) 단위로 책임이 나뉘어 있어 누가 무엇을 했는지 추적됩니다. 공통 영역은 합의 후 수정한다는 규칙도 함께 말합니다.
:::

:::details "남의 도메인은 얼마나 이해하나요?"
전 도메인이 같은 4계층 골격과 횡단 규칙을 공유하므로, 구조 수준에서는 모두 설명할 수 있습니다. 다만 세부 구현 깊이는 담당자가 더 정확하다고 정직하게 말하고, 제가 아는 범위(인터페이스·테이블·역할)까지만 단정합니다. 모르는 부분을 아는 척하지 않습니다.
:::

:::details "AI를 직접 붙인 경험이 있나요? 핵심 난점은?"
외부 모델은 응답 형식이 흔들리고 가끔 실패한다는 점이 난점입니다. 형식은 구조화 출력(GPT는 strict JSON Schema, Gemini는 구조화 JSON)으로 강제하고, 실패는 폴백(추천이면 DB 캐시 → 트렌딩)으로 사용자 경험을 지켰습니다. 다만 응답 품질을 정량으로 평가하는 체계는 아직 없어 향후 과제로 둡니다.
:::

:::details "이 프로젝트의 가장 큰 약점은?"
항공권이 실제 외부 API가 아니라 MockFlightOfferProvider라는 점, AI 응답 품질 정량 평가가 없다는 점, 화면이 데스크톱 JSP 위주라 모바일 반응형과 Swagger가 부재한 점입니다. 항공권은 인터페이스로 추상화돼 있어 실제 어댑터만 갈아끼우면 되도록 설계해 둔 점을 함께 말해 한계와 대비를 동시에 보여줍니다.
:::

## 8. 직접 말해보기

녹음하고 시간을 재면서 아래를 막힘 없이 답할 수 있으면 합격선이다.

- 30초 안에 TripTogether가 무슨 서비스이고 어떤 스택인지 말해 보라.
- 1분 안에 한 요청이 인터셉터 → AOP → 서비스 → 매퍼 → DB → 화면/알림으로 도는 길을 풀어 보라.
- 본인 담당 도메인을 일반 화법으로 소개하고, 직접 해결한 문제 하나를 1분간 깊게 설명해 보라.
- 기술 선택 3가지(MyBatis·JSP·다중 AI)를 각각 트레이드오프와 함께 말해 보라.
- 가장 어려웠던 점 하나를 문제 → 해결 → 배운 점 순서로 1분간 말해 보라.
- 프로젝트의 한계 한 가지와 구체적 개선안을 정직하게 제시해 보라.
