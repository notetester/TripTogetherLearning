# 관리자·운영 면접 플레이북

> 관리자·운영 도메인은 데이터를 소유하지 않고 조회·조치·추적만 하는 횡단 거버넌스 계층이다. 이 페이지는 1분/3분 답변, 핵심 기술 선택의 근거, 예상 질문 10개 이상을 한 곳에 압축한다.

## 1. 한 줄 정의

TripTogether의 관리자·운영 도메인은 각 도메인이 생성한 데이터를 운영자가 한곳에서 **관측·조치·추적·통제**하게 하는 거버넌스 계층이며, 모든 조치를 표준 사유 코드로 감사 로그에 남기고, 권한은 그룹으로 분리하고, 정책·설정은 DB 우선으로 재배포 없이 바꾸는 구조다.

이 자료는 **중립 서술**을 원칙으로 한다. TripTogether는 4명이 도메인을 수직 분담해 만든 팀 프로젝트이며, 관리자·운영은 그중 한 담당 영역이다. 코드·DB·ADR에 드러난 사실만 기준으로 하며 특정 개인을 내 영역으로 지칭하지 않는다.

## 2. 1분 / 3분 답변 스크립트

### 1분 버전 (엘리베이터 피치)

:::tip 그대로 말해보기
"관리자·운영 도메인은 커뮤니티 글, 문의, 결제, AI 챗봇 같은 각 도메인이 만든 데이터를 운영자가 한 화면에서 보고 조치하는 횡단 계층입니다. `/admin` 과 최상위 `/superAdmin` 경로로 나뉘고, 인터셉터 체인과 AOP `@RequireAdmin` 으로 보호됩니다. 네 가지 책임이 핵심입니다. 첫째 관측으로 대시보드 통계와 회원 360 뷰, 둘째 조치로 게시글·회원·IP 차단, 셋째 추적으로 모든 관리자 조치를 표준 사유 코드와 함께 `ADMIN_ACTION_AUDIT` 에 남기는 것, 넷째 통제로 권한을 그룹으로 분리하고 운영 설정을 DB 우선으로 바꾸는 것입니다. 설계 철학을 한 줄로 줄이면 자동 차단이 아니라 사람이 최종 판단하고, 모든 결정을 추적 가능하게 남긴다는 것입니다."
:::

### 3분 버전 (설계 결정까지)

3분 답변은 1분 버전에 **왜 그렇게 설계했나**를 네 갈래로 덧붙인다.

1. **왜 데이터를 소유하지 않고 조치만 하나** — 게시글 본문은 커뮤니티 도메인이 소유하고, 운영 화면은 그 글을 BLUR 처리하거나 신고를 종결하는 조치 권한만 가진다. 이 분리가 소프트 삭제와 감사 로그 설계의 전제다. 운영이 데이터 소유까지 가져가면 도메인 경계가 무너지고 책임이 흐려진다.
2. **왜 감사 로그를 표준 사유 코드로 남기나** — 관리자 권한은 회원 차단·콘텐츠 삭제처럼 되돌리기 어려운 영향을 준다. 자유 텍스트만 남기면 사후 집계·필터·다국어 표기가 불가능하다. 그래서 `reason_code` 를 1급 시민으로 올리고 가변 인자는 `reason_args` JSON 으로 둔다.
3. **왜 자동 차단을 금지했나(ADR-0001, ADR-0010)** — Perspective 독성 점수나 신고 누적만으로 즉시 삭제하면 거짓 양성 비용이 크다. 그래서 자동은 BLUR(점진 공개 차단)까지만 하고, 최종 해제는 관리자가 한다. Human-in-the-Loop이다.
4. **왜 권한 그룹과 런타임 설정인가** — 권한을 개별 부여하면 관리자 수만큼 관리 비용이 커지므로 정책과 부여를 테이블로 분리했고, 운영 임계값은 재배포 없이 바꿔야 하므로 DB 우선 설정으로 외부화했다.

## 3. 기술 선택의 근거 (왜 이렇게)

면접에서 가장 깊게 파고드는 네 가지 결정이다. 각 항목은 별도 학습 페이지로 연결된다.

### 왜 런타임 설정을 DB 우선으로 했나

운영 중 자주 바뀌는 값(모더레이션 임계치, 기능 토글 등)을 코드 상수나 properties 에 두면 변경마다 재배포가 필요하다. `RuntimeSettingService` 가 `APPLICATION_RUNTIME_SETTING` 을 읽고, 변경은 `APPLICATION_RUNTIME_SETTING_HISTORY` 에 이력으로 남는다.

```text
RuntimeSettingVO {
  settingKey, settingGroup, settingValue, fallbackValue,
  valueType,   // 문자/숫자/불리언 등 타입 검증
  secret,      // 시크릿 여부 → 화면에서 마스킹
  editable, active
}
```

핵심은 세 가지다. `fallbackValue` 로 DB 미설정 시 안전한 기본값을 보장하고, `valueType` 으로 잘못된 타입 주입을 막고, `secret` 플래그로 화면 노출을 통제한다. `AdminRuntimeSettingController` 가 조회·수정 화면을 제공한다.

:::warning DB 우선의 함정과 답변
"DB 우선이면 DB 장애 시 어떻게 되나"라는 꼬리질문이 자주 온다. 답은 `fallbackValue` 다. 설정을 못 읽으면 코드에 내장된 안전 기본값으로 동작하므로, DB 의존이 단일 장애점이 되지 않는다.
:::

### 왜 감사 로그를 분리하고 사유 코드를 쓰나

하나의 거대한 로그 테이블은 쓰기 경합과 질의 비용이 커지고 보존 정책·접근 권한을 차등화하기 어렵다. 그래서 책임 범위별로 테이블을 분리했다.

| 책임 | 테이블 | 성격 |
| --- | --- | --- |
| 관리자 의도적 조치 | `ADMIN_ACTION_AUDIT` | 좁게, 사유 코드와 함께 |
| 보안 민감 이벤트 | `SECURITY_ACTION_AUDIT` | 계정 복구·차단 등 |
| 급여 변경 | `SALARY_CHANGE_AUDIT` | superAdmin 영역 |
| 일반 활동 텔레메트리 | `USER_ACTIVITY_LOG` | 넓게, 자동 기록 |

`AdminActionAuditService.record(...)` 가 단일 진입점이고 `AdminActionAuditVO` 가 적재된다. `reason_code` 로 사유를 표준화하면 빈도 집계·필터·다국어 라벨이 자연스럽게 따라온다. 사용자 외래키는 `ON DELETE SET NULL` 이라 관리자 계정이 삭제돼도 감사 행은 보존된다.

### 왜 자동 차단을 금지했나

자동 차단 금지는 단순 정책이 아니라 ADR로 문서화된 결정이다(ADR-0001 신고 자동차단 금지, ADR-0010 AI 모더레이션 사람 승인).

```text
콘텐츠 위험 신호                자동 조치           최종 조치
─────────────────────────────────────────────────────────
Perspective 독성 점수 높음   → ai_flagged + BLUR → 관리자 해제/삭제
신고 누적(3-strike)          → 누적 BLUR         → 관리자 검토
명백한 위반                  → (없음)            → 관리자 즉시 조치
```

거짓 양성(정상 글을 위반으로 오판)의 비용이 거짓 음성보다 사용자 신뢰에 더 치명적이라는 판단이다. 그래서 AI/신고는 가시성을 낮추는 BLUR 까지만 하고, 영구 조치는 사람이 한다.

### 왜 권한을 그룹으로 분리했나

관리자에게 권한을 개별 부여하지 않고 정책(무엇을 할 수 있나)과 부여(누구에게)를 테이블로 분리했다.

| 테이블 | 역할 |
| --- | --- |
| `ADMIN_PERMISSION` / `ADMIN_PERMISSION_POLICY` | 개별 권한 정의 |
| `ADMIN_PERMISSION_GROUP_POLICY` | 권한 묶음(역할) 정의 |
| `ADMIN_PERMISSION_GROUP_ITEM` | 그룹에 속한 권한 매핑 |
| `ADMIN_PERMISSION_GROUP` | 특정 관리자에게 그룹 부여 |
| `ADMIN_PERMISSION_CODE_POLICY` 외 | 실효 권한 코드 확장 |
| `ADMIN_EFFECTIVE_PERMISSION_VW` | 최종 실효 권한 조회 뷰 |

정책과 부여를 분리했기 때문에 그룹 구성만 바꾸면 그 그룹에 속한 모든 관리자의 실효 권한이 한 번에 바뀐다. 최종 권한은 매번 계산하지 않고 `ADMIN_EFFECTIVE_PERMISSION_VW` 뷰로 평탄화해 조회한다.

## 4. 운영 화면 공통 패턴 (한 번 익히면 전부 설명됨)

회원·문의·신고·로그인 감사 등 도메인이 달라도 목록 화면은 한 가지 틀을 공유한다.

```text
검색 VO(다중필터+정렬+화이트리스트)
  → Service(정규화·집계)
  → Mapper(동적 SQL)
  → 페이징 VO(블록 계산)
  → JSP 행 프래그먼트
  → Excel 내보내기(Apache POI)
```

`AdminSearchVO` 같은 검색 VO의 getter에서 정렬 컬럼·상태 필터를 **화이트리스트로 1차 정규화**한다. 외부 요청값을 그대로 ORDER BY 에 넣으면 SQL 인젝션·오동작 위험이 있기 때문이다. 면접에서 "운영 목록이 많은데 어떻게 일관성을 유지했나"를 물으면 이 공통 패턴을 답하면 된다.

## 6. 면접 답변 3단계 (결론 → 근거 → 트레이드오프)

어떤 질문이 와도 이 3단계 틀로 답하면 밀도가 높아진다.

1. **결론 먼저** — "운영 도메인의 설계 원칙은 자동 차단 금지와 전수 추적입니다."
2. **근거(코드/테이블/ADR)** — "조치는 `AdminActionAuditService.record` 한 곳에서 `ADMIN_ACTION_AUDIT` 에 사유 코드와 함께 남고, 모더레이션은 ADR-0010 에 따라 BLUR 후 관리자 해제 구조입니다."
3. **트레이드오프** — "이 구조는 즉각 차단보다 운영 부하가 큽니다. 대신 거짓 양성으로 정상 사용자를 막는 비용을 피했고, 분쟁 시 책임 추적이 가능합니다."

## 7. 예상 질문 + 모범 답안

:::details Q1. 관리자가 게시글을 가렸을 때 어떻게 추적하나요?
조치를 수행하는 컨트롤러가 `AdminActionAuditService.record(...)` 를 명시적으로 호출해 `ADMIN_ACTION_AUDIT` 에 한 줄을 남깁니다. who(actor_user_idx)·when(created_at)·what(action_type/target)·why(reason_code/reason_args)가 한 행에 들어갑니다. 사유는 자유 텍스트가 아니라 표준 코드라 사후 집계와 다국어 라벨이 가능합니다. 관리자 계정이 삭제돼도 외래키가 ON DELETE SET NULL 이라 감사 행은 보존됩니다.
:::

:::details Q2. AI가 독성으로 표시한 글을 왜 바로 삭제하지 않나요?
거짓 양성 비용 때문입니다. Perspective 독성 점수나 신고 누적은 비동기로 평가되고, 자동 조치는 가시성을 낮추는 BLUR 까지만 합니다. 영구 삭제 같은 되돌리기 어려운 조치는 관리자가 최종 판단합니다. ADR-0001 과 ADR-0010 에 이 Human-in-the-Loop 원칙이 문서화돼 있습니다. 정상 글을 잘못 막아 사용자 신뢰를 잃는 비용이, 위반 글이 잠깐 더 노출되는 비용보다 크다고 판단했습니다.
:::

:::details Q3. 운영 중 모더레이션 임계값을 바꿔야 하면 재배포하나요?
아니요. `APPLICATION_RUNTIME_SETTING` 에서 DB 우선으로 읽습니다. `AdminRuntimeSettingController` 화면에서 값을 바꾸면 즉시 반영되고 `APPLICATION_RUNTIME_SETTING_HISTORY` 에 변경 이력이 남습니다. DB 미설정이나 장애 시에는 `fallbackValue` 라는 코드 내장 기본값으로 동작해 단일 장애점이 되지 않습니다. `valueType` 으로 타입을, `secret` 으로 화면 마스킹을 제어합니다.
:::

:::details Q4. 관리자마다 권한을 어떻게 다르게 주나요?
권한 그룹으로 정책과 부여를 분리합니다. 개별 권한은 `ADMIN_PERMISSION_POLICY` 에, 묶음(역할)은 `ADMIN_PERMISSION_GROUP_POLICY` 에 정의하고, 관리자에게는 그룹을 부여합니다. 그룹 구성만 바꾸면 그 그룹의 모든 관리자 권한이 한 번에 바뀝니다. 매 요청마다 권한을 계산하지 않고 `ADMIN_EFFECTIVE_PERMISSION_VW` 뷰로 실효 권한을 평탄화해 조회합니다. 권한 검사 자체는 인터셉터 체인과 AOP @RequireAdmin 으로 가로챕니다.
:::

:::details Q5. 회원 한 명의 모든 활동을 어떻게 한 화면에 모으나요?
회원 360 뷰는 user_idx 를 축으로 여러 도메인 데이터(가입·로그인 이력·커뮤니티·문의·신고·지갑)를 집계합니다. 각 도메인 데이터를 운영이 소유하지 않고 조회만 하므로, 조인과 도메인별 카운트 쿼리를 모아 한 모델로 렌더링합니다. 성능은 조회 패턴에 맞춘 인덱스와 필요한 만큼만 페이징·지연 조회하는 식으로 다룹니다.
:::

:::details Q6. IP 차단은 단일 IP만 막나요?
아니요. `IP_BLOCKLIST` 규칙은 SINGLE_IP, CIDR, RANGE 세 가지 매치 타입을 지원합니다. CIDR 표기(예: 대역 단위)와 시작·끝 IP 범위까지 막을 수 있고, `IpBlockInterceptor` 가 요청 단계에서 평가합니다. 대량 작업은 `IP_BLOCK_BATCH` 와 배치 오퍼레이션 테이블로 관리해 일괄 추가·해제와 그 이력을 남깁니다.
:::

:::details Q7. 자유 텍스트 대신 사유 코드를 쓰면 사람이 읽기 불편하지 않나요?
둘 다 남깁니다. 기계 처리용으로 `reason_code` 와 `reason_args` JSON 을, 사람용으로 `detail_summary` 요약 문장을 함께 기록합니다. 코드는 집계·필터·다국어를, 요약은 가독성을 담당합니다. 즉 표준화와 가독성을 분리해 둘 다 얻습니다.
:::

:::details Q8. 모든 관리자 조치를 인터셉터로 자동 기록하지 않은 이유는?
일반 활동은 `ActivityLogInterceptor` 가 자동으로 넓게 기록합니다. 하지만 관리자 조치는 의도적 결정이라 무엇을 왜 했는지 사유 코드가 필요한데, 인터셉터는 비즈니스 의미를 모릅니다. 그래서 조치 컨트롤러가 명시적으로 `record(...)` 를 호출합니다. 자동 텔레메트리와 의도적 감사 로그를 의도적으로 분리한 것입니다.
:::

:::details Q9. superAdmin 과 admin 은 어떻게 다른가요?
admin 은 콘텐츠·회원 운영(`/admin`)을, superAdmin 은 관리자 계정·조직 구조·급여(`/superAdmin`)를 다룹니다. 조직 구조는 position/rank/tier 같은 다수의 enum 으로 표현하고, 급여 변경은 `SALARY_CHANGE_AUDIT` 로 별도 감사하며 Excel 업로드를 지원합니다. 인터셉터 체인에서 admin → superAdmin 순으로 권한을 차등 검사합니다.
:::

:::details Q10. 운영 화면이 도메인마다 많은데 일관성을 어떻게 유지했나요?
목록 화면은 검색 VO(다중필터+정렬+화이트리스트) → 서비스(정규화·집계) → 매퍼(동적 SQL) → 페이징 VO → JSP 행 프래그먼트 → Excel 내보내기라는 공통 패턴을 공유합니다. 특히 정렬·필터 값은 검색 VO getter 에서 화이트리스트로 정규화해 외부 입력이 SQL 에 직접 흘러가지 않게 합니다. 도메인이 달라도 같은 틀이라 유지보수 비용이 낮습니다.
:::

:::details Q11. AI 챗봇/도우미는 운영에서 어떻게 통제하나요?
사용량·쿼터·차단 중심입니다. 등급별 일일 쿼터를 두고 초과 시 막으며, 부적절 사용은 차단 목록으로 관리합니다. 모더레이션 스케줄러가 누적 신호를 주기적으로 점검합니다. 다만 AI 응답 품질 자체를 정량 평가하는 메트릭은 아직 없어, 이 부분은 솔직히 향후 과제라고 말합니다.
:::

:::details Q12. 매출 통계가 실제 매출인가요?
대부분 실거래 기반이지만 정직하게 구분합니다. 결제는 Toss 연동이고 환불은 환불 로그로 추적되지만, 항공권 공급은 Mock 프로바이더라 항공 부분 매출은 실거래가 아닙니다. 면접에서는 이 한계를 먼저 밝히는 편이 신뢰를 줍니다.
:::

## 8. 직접 말해보기

아래를 보지 않고 소리 내어 답해보면 체화된다.

1. 관리자 도메인의 네 가지 책임을 한 문장씩 말해보라(관측·조치·추적·통제).
2. 자동 차단을 금지한 이유를 거짓 양성 비용 관점에서 30초로 설명하라.
3. 런타임 설정 DB 우선의 장점과, DB 장애 시 대비책(`fallbackValue`)을 함께 말하라.
4. 권한 그룹에서 정책과 부여를 분리하면 무엇이 쉬워지는지 한 문장으로 말하라.
5. 감사 로그에서 사유 코드와 요약 문장을 둘 다 남기는 이유를 말하라.
