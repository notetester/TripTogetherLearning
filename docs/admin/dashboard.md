# 관리자 대시보드

> 운영자가 로그인 직후 처음 만나는 화면. 회원·로그인·콘텐츠·매출 지표를 한 페이지에 모아 보여주고, 거기서 회원 목록 같은 운영 검색 화면으로 진입한다.

이 페이지는 TripTogether의 관리자 도메인 중 통합 운영 대시보드와, 그 진입점에서 이어지는 다중필터 검색·정렬·페이징 패턴을 다룬다. 관리자 도메인은 4인이 도메인을 나눠 만든 팀 프로젝트의 한 축이며, 대시보드는 여러 도메인(회원·커뮤니티·문의·신고·매출)의 집계를 모아 운영자에게 전달하는 크로스도메인 관문 역할을 한다.

## 1. 한 줄 정의

관리자 대시보드는 운영에 필요한 핵심 지표를 한 번의 서버 렌더링으로 모아 보여주는 요약 화면이며, 회원/문의/신고 등 세부 운영 목록 화면은 모두 다중필터 + 정렬 + 페이징 + 내보내기라는 동일한 패턴을 공유한다.

## 2. 왜 이렇게 설계했나

- **운영자는 숫자를 먼저 본다.** 신규 가입, 오늘 로그인 성공/실패, 대기 중인 문의·신고처럼 즉시 대응이 필요한 신호를 첫 화면에서 한눈에 파악해야 한다. 그래서 대시보드는 상세 목록이 아니라 카운트와 추세 차트 중심으로 구성한다.
- **SSR(JSP) 기반의 단순함.** 프런트가 JSP(JSTL/EL)라 대시보드 본체는 컨트롤러가 Model에 통계 객체를 담아 한 번에 렌더링한다. 차트 데이터처럼 사용자가 기간을 바꿔 다시 조회하는 부분만 별도 JSON 엔드포인트로 뺐다.
- **목록 화면 패턴 통일.** 회원·문의·신고·로그인 감사 등 서로 다른 도메인의 운영 화면이 제각각이면 유지보수가 어렵다. 그래서 검색 파라미터 VO + 페이징 VO + 행 프래그먼트라는 한 가지 틀을 공유한다.
- **외부 입력 신뢰 금지.** 정렬 컬럼이나 상태 필터를 외부 요청값 그대로 SQL ORDER BY에 넣으면 SQL 인젝션·오동작 위험이 있다. 그래서 검색 VO의 getter에서 화이트리스트로 1차 정규화한다.

## 3. 어떤 기술로 구현했나 (실제 클래스·테이블)

전형적인 4계층(controller to service to mapper to vo)으로 구현돼 있다.

| 계층 | 클래스 / 파일 | 역할 |
| --- | --- | --- |
| Controller | `AdminController` (`/admin`) | 대시보드 렌더링, 매출 통계 JSON, 회원 목록·프래그먼트·내보내기 |
| Service | `AdminService` / `AdminServiceImpl` | 통계 집계, 차트 시계열 채우기, 검색 정규화 |
| Mapper | `AdminMapper` + `adminMapper.xml` | 통계·일자별 집계·회원 검색 쿼리 |
| 통계 VO | `AdminStatsVO` | 회원/로그인/소셜연동/커뮤니티/문의/신고 요약 카운트 |
| 차트 VO | `AdminDashboardChartVO` | 일자 labels + 신규가입·로그인성공·로그인실패 시계열 |
| 매출 VO | `AdminSalesDailyStatVO` | 일자별 총거래·현금성매출·마일리지·취소 집계 |
| 검색 VO | `AdminSearchVO` | 회원 목록 다중필터·정렬·페이징 파라미터(+화이트리스트) |
| 페이징 VO | `AdminPageVO` | 화면 바인딩용 페이지 블록 계산 |

대시보드 진입 메서드는 다음 한 곳에 모여 있다.

```java
@GetMapping({"", "/"})
public String dashboard(Model model) {
    model.addAttribute("stats", adminService.getStats());          // 요약 카운트
    model.addAttribute("chart", adminService.getDashboardChart(7)); // 7일 추세
    model.addAttribute("salesStats", adminService.getSalesDailyStats(30)); // 30일 매출
    model.addAttribute("activeMenu", "dashboard");
    return "admin/dashboard";
}
```

대시보드 접근 권한은 컨트롤러 코드가 아니라 인터셉터 체인(admin 인터셉터)과 AOP 권한(@RequireAdmin)으로 가로채는 구조라, 컨트롤러 메서드 자체는 비즈니스 로직에 집중한다. 관리자 권한 모델 자체는 별도 페이지(권한 그룹)에서 다룬다.

## 4. 동작 원리 (흐름·표·작은 코드)

### 4-1. 요약 카운트

`getStats()`는 매퍼의 단일 집계 쿼리 결과를 `AdminStatsVO` 한 개로 받는다. 담는 지표는 다음과 같다.

- 회원: 전체/활성/휴면/탈퇴, 오늘 신규 가입
- 로그인: 오늘 로그인 성공, 오늘 로그인 실패, 오늘 로그아웃(로컬·소셜 제공자별 분리)
- 소셜 연동: 카카오/네이버/구글 연동 회원 수
- 콘텐츠/운영: 전체·활성 커뮤니티 게시글, 미처리 신고, 전체·대기·완료 문의

상태 기반 카운트는 소프트삭제 패턴과 맞물린다. 예를 들어 활성 게시글은 post_status = ACTIVE, 탈퇴 회원은 account_status = DELETED 기준으로 센다. 즉 물리 삭제가 아니라 상태 컬럼으로 구분하므로, 같은 테이블에서 활성·삭제 카운트를 동시에 뽑을 수 있다.

### 4-2. 추세 차트 (빈 날짜 채우기)

DB는 거래·로그인이 발생한 날짜만 행으로 돌려준다. 하지만 차트는 7일 전부가 연속으로 채워져야 하므로, 서비스가 시작일부터 하루씩 순회하며 비는 날을 0으로 메운다.

```java
LocalDate start = today.minusDays(spanParam);
for (int i = 0; i < days; i++) {
    LocalDate d = start.plusDays(i);
    String key = d.format(keyFmt);             // yyyy-MM-dd
    labels.add(d.format(labelFmt));            // MM/dd 라벨
    newMembers.add(nmMap.getOrDefault(key, 0L)); // 없는 날은 0
    ...
}
```

이렇게 DB Map을 미리 만들어 날짜 키로 조회하면, 데이터가 없는 날도 그래프가 끊기지 않고 연속 시계열로 그려진다.

### 4-3. 매출 통계 (시뮬레이션 정직 분리)

`AdminSalesDailyStatVO`는 총 거래 규모(grossSales)와 실제 현금성 매출(cashSales)을 분리한다. 결제가 실제 PG 정산이 아니라 캐시·마일리지 차감 시뮬레이션이기 때문에, 마일리지로 결제된 금액을 현금 매출로 착시하지 않도록 의도적으로 컬럼을 나눴다. 기간은 매출 통계 전용 JSON 엔드포인트(/admin/sales/stats?days=)로 다시 조회한다.

### 4-4. 다중필터 목록 패턴 (회원 목록 예시)

대시보드에서 회원 목록으로 들어가면 동일한 운영 패턴이 반복된다.

```text
요청 파라미터(AdminSearchVO)
  → getter 화이트리스트 정규화(상태/권한/제공자/정렬컬럼/정렬방향/페이지크기)
  → countMembers (총 건수)
  → AdminPageVO.of (페이지 블록 계산, 페이지 초과 시 마지막 페이지로 보정)
  → findMembers (한 페이지)
  → JSP 렌더 또는 _memberRowsFragment(부분 갱신)
```

핵심 안전장치는 화이트리스트다. 정렬 컬럼은 허용 집합에 없으면 createdAt으로, 페이지 크기는 10/20/50/100이 아니면 20으로 강제된다.

```java
public String getSortBy() {
    String value = normalizeText(sortBy);
    return value != null && VALID_SORT.contains(value) ? value : "createdAt";
}
```

같은 화면이 세 가지 응답을 제공한다: 전체 페이지(JSP), 행 프래그먼트(검색 시 표 부분만 교체), JSON API. 프래그먼트 응답에는 X-Section-Total/Page/Size/Pages 같은 헤더로 페이징 메타를 함께 내려, 프런트가 페이지네이션을 다시 그릴 수 있게 한다. 검색 결과를 CSV·Excel(Apache POI)로 내보내는 경로도 같은 검색 조건을 재사용한다.

:::tip 정렬 화이트리스트가 핵심
다중필터 검색에서 가장 흔한 실수는 정렬 컬럼을 요청값 그대로 ORDER BY에 넣는 것이다. TripTogether는 검색 VO getter 단계에서 허용 목록과 대조해 보정하므로, 부정한 정렬 키가 매퍼까지 도달하지 못한다.
:::

## 6. 면접 답변 3단계

1. **한 줄:** 관리자 대시보드는 회원·로그인·콘텐츠·매출 핵심 지표를 한 화면에 모아 보여주는 운영 관문이고, 거기서 이어지는 운영 목록은 다중필터·정렬·페이징·내보내기를 공통 패턴으로 공유합니다.
2. **설계 의도:** SSR(JSP) 기반이라 요약·차트·매출은 컨트롤러가 Model에 한 번에 담아 렌더링하고, 기간을 바꿔 다시 보는 차트만 JSON 엔드포인트로 분리했습니다. 빈 날짜를 0으로 채워 연속 시계열을 만드는 처리, 그리고 매출에서 총거래와 현금성 매출을 분리해 시뮬레이션 결제를 매출로 착시하지 않게 한 점이 포인트입니다.
3. **안전·확장:** 검색 VO getter에서 정렬 컬럼·상태·페이지 크기를 화이트리스트로 정규화해 외부 입력을 매퍼 전에 차단합니다. 목록 화면은 전체 페이지·행 프래그먼트·JSON 세 응답을 같은 검색 조건으로 제공해, 부분 갱신과 내보내기까지 한 흐름으로 처리합니다.

## 7. 꼬리질문 + 모범답안

:::details 차트에서 데이터가 없는 날은 어떻게 처리하나요
DB는 발생한 날짜만 행으로 돌려줍니다. 서비스에서 결과를 날짜 키 Map으로 만든 뒤, 시작일부터 하루씩 순회하며 Map에 없는 날은 0을 채웁니다. 이렇게 하면 7일이면 항상 7개 라벨과 7개 값이 나와 그래프가 끊기지 않습니다.
:::

:::details 매출 통계에서 총거래와 현금성 매출을 왜 나눴나요
결제가 실제 PG 정산이 아니라 캐시·마일리지 차감 시뮬레이션이기 때문입니다. 마일리지로 차감된 금액까지 현금 매출로 합치면 실제 현금 유입을 과대평가하게 됩니다. 그래서 grossSales(총거래 규모)와 cashSales(현금성)를 분리하고, 취소 금액·취소 건수도 별도로 집계합니다.
:::

:::details 목록 정렬을 사용자가 지정하는데 SQL 인젝션은 어떻게 막나요
정렬 컬럼·방향을 요청값 그대로 쓰지 않습니다. AdminSearchVO의 getter가 허용된 정렬 키 집합과 대조해 일치하지 않으면 createdAt으로, 정렬 방향도 ASC가 아니면 DESC로 강제합니다. 페이지 크기도 정해진 값만 허용합니다. 즉 매퍼에 도달하기 전에 값이 정규화됩니다.
:::

:::details 검색했더니 현재 페이지가 총 페이지를 넘으면 어떻게 되나요
먼저 총 건수를 센 뒤 AdminPageVO에서 총 페이지를 계산하고, 요청 페이지가 총 페이지를 초과하면 마지막 페이지로 보정합니다. 그 보정된 페이지로 다시 목록을 조회하므로, 필터를 좁혀 결과가 줄어도 빈 페이지가 뜨지 않습니다.
:::

:::details 검색 결과를 부분만 갱신한다는데 어떤 구조인가요
같은 회원 목록 데이터를 전체 페이지·행 프래그먼트·JSON 세 형태로 제공합니다. 검색·정렬을 바꾸면 표 영역만 행 프래그먼트로 교체하고, 응답 헤더에 총건수·현재페이지·페이지수 같은 페이징 메타를 실어 보내 프런트가 페이지네이션을 다시 그립니다. 전체 새로고침 없이 표만 갱신되는 운영 편의 패턴입니다.
:::

## 8. 직접 말해보기

- 대시보드가 한 화면에 모으는 지표 카테고리 네 가지(회원·로그인·콘텐츠·매출)를 들고, 각각 운영자가 그 숫자로 무엇을 결정하는지 한 문장씩 설명해 보세요.
- 차트의 빈 날짜 채우기와 목록의 페이지 초과 보정은 둘 다 빈 결과를 다루는 처리입니다. 왜 두 곳 모두 필요한지 사용자 경험 관점에서 말해 보세요.
- 매출 통계가 시뮬레이션이라는 한계를 면접에서 정직하게 말하면서도, 그 한계를 의식해 컬럼을 분리한 설계를 강점으로 연결해 보세요.
