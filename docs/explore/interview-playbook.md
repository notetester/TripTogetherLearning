# 탐색·커머스 면접 플레이북

> 탐색(개인화 추천)부터 커머스(항공권·결제·지갑·패키지)까지, 외부 의존을 끊고도 무너지지 않게 설계한 이유를 면접에서 30초·3분으로 풀어내는 법.

이 페이지는 개념 설명이 아니라 **말하기 대본**이다. 같은 도메인의 세부 페이지([탐색 필터](/explore/explore-filters), [AI 추천](/explore/ai-recommendation-gemini), [추천 캐시·폴백](/explore/recommendation-cache-fallback), [항공권 Mock](/explore/flight-mock), [Toss 결제](/explore/toss-payments), [3원 지갑](/explore/three-wallet), [패키지 마켓플레이스](/explore/package-marketplace))를 이미 읽었다고 가정하고, 그 내용을 면접관 앞에서 압축해 전달하는 데 집중한다.

::: tip 전체 맥락이 필요하면
[도메인 전체 개요](/domains) · [담당별 보기](/by-area/) · [전체 흐름](/flow/) 으로 이 도메인이 사용자 여정(탐색→계획→예약→공유)의 어디에 붙는지 먼저 잡아라.
:::

## 1. 한 줄 정의

이 도메인은 **여행지를 개인화로 추천하고(탐색), 그 여행지를 실제 구매·예약 흐름으로 잇는(커머스) 두 축**을 다루며, 두 축 모두 **외부 의존(AI·결제·항공 API)이 실패해도 사용자 경험이 끊기지 않도록 폴백과 추상화를 깔아둔 것**이 핵심이다.

## 2. 1분 답변 (엘리베이터 피치)

면접관이 "이 부분을 설명해 보세요"라고 했을 때 그대로 말할 한 덩어리.

> TripTogether에서 저희 팀이 다룬 탐색·커머스 영역은 크게 두 가지입니다.
> 하나는 여행지 **개인화 추천**입니다. 사용자가 여행지 상세를 보며 머문 체류 시간을 `SPOT_VIEW_LOG`에 기록하고, 최근 30건의 체류·방문 빈도에 가중치를 줘서 관심 태그를 뽑은 뒤 Gemini 2.5 Flash로 다음에 흥미를 느낄 여행지 3개를 추천합니다. 핵심은 **DB 캐시 → Gemini → 트렌딩의 3단 폴백**이라, AI가 죽어도 추천 영역이 비지 않습니다.
> 다른 하나는 **커머스**입니다. 항공권은 실제 외부 API 대신 `FlightOfferProvider` 인터페이스 뒤의 Mock 구현체로 견적을 생성하고, 결제는 Toss Payments로 캐시를 충전한 뒤 **캐시·마일리지·포인트 3원 지갑**에서 차감합니다. 항공권은 캐시와 마일리지를 섞어 결제하되 마일리지는 금액의 30%로 제한했고, 모든 변동은 `USER_WALLET_HISTORY` 원장에 잔액과 함께 남깁니다.
> 한 문장으로 요약하면, **외부 의존이 흔들려도 서비스가 멈추지 않도록 폴백·추상화·원장**을 깐 영역입니다.

## 3. 3분 답변 (구조를 보여주는 버전)

1분 버전을 듣고 "더 자세히"가 들어오면, 아래 4블록을 순서대로.

**(1) 탐색 — 데이터를 모으는 단계**
여행지 탐색은 `ExploreController`가 7개 탭 필터(지역·테마 등)로 `SPOT`을 조회하는 데서 시작한다. 사용자가 여행지 상세를 보면 체류 시간이 `SPOT_VIEW_LOG`(spot_idx, stay_seconds)에 쌓이고, 찜은 `SPOT_FAVORITE`, 좋아요는 `SPOT_LIKE`로 분리해 저장한다. 이 로그가 추천의 입력이 된다.

**(2) 추천 — AI를 쓰되 의존하지 않는 단계**
`RecommendService`가 최근 30건 로그를 모아 **체류 시간 + 방문 빈도에 가중치**를 줘 관심 태그를 추출하고, 후보 여행지에 `tag_match_score`를 매겨 정렬한 뒤 Gemini에 "이 후보 중 3개만, 이전 추천·최근 방문은 제외"라는 프롬프트를 보낸다. 응답은 JSON 배열로 받아 잘린 객체까지 복구해 파싱하고, FK 검증 후 `SPOT_RECOMMEND`에 캐시한다.

**(3) 커머스 — 외부 경계를 인터페이스로 막는 단계**
항공권은 `FlightOfferProvider` 인터페이스만 서비스가 알고, 실제 구현은 `MockFlightOfferProvider`다. 결제는 Toss로 캐시를 충전(`prepareTossCharge` → `completeTossCharge`)한 뒤, 항공권 결제 시 캐시+마일리지 혼합으로 차감한다.

**(4) 정합성 — 돈을 다루니 원장으로 증명**
지갑은 `USERS`의 `cash_balance`·`mileage_balance`·`point_balance` 3개 잔액 컬럼이고, 모든 증감은 `USER_WALLET_HISTORY`(asset_type, change_type, amount, **balance_after**)에 변동 후 잔액과 함께 기록한다. 환불은 같은 원장에 REFUND로 역분개하고 `WALLET_REFUND_LOG`에 따로 남겨, "지금 잔액"과 "변동 내역"이 항상 맞도록 했다.

## 4. 기술 선택의 근거 (면접 핵심)

면접관이 진짜 알고 싶은 건 "어떻게"보다 **"왜 그렇게 골랐나"**다. 세 가지 선택을 방어 논리와 함께 외워라.

| 선택 | 왜 이렇게 했나 | 안 했으면 무슨 문제 |
| --- | --- | --- |
| **3단 폴백** (DB캐시 → Gemini → 트렌딩) | AI는 느리고 비싸고 가끔 실패한다. 캐시로 비용·지연을 줄이고, 실패 시 트렌딩으로 빈 화면을 막는다 | AI 한 번 죽으면 추천 영역 전체가 빈칸 → 핵심 동선 붕괴 |
| **3원 지갑** (캐시·마일리지·포인트 분리) | 충전한 돈(캐시), 보상으로 준 돈(마일리지), 활동 보상(포인트)은 환불·만료·사용처 규칙이 다르다 | 하나로 합치면 환불 정산·회계 추적·악용 방지가 전부 꼬인다 |
| **Mock 프로바이더 추상화** (`FlightOfferProvider`) | 외부 항공 API 키가 없는 상태에서도 구매 플로우 전체를 끝까지 만들어야 했다 | Mock 코드가 서비스에 박히면, 실 API 도입 시 결제·화면까지 다 뜯어야 한다 |

::: details 왜 3단 폴백인가 — 한 단계 더
"AI 추천"은 들어가는 비용이 호출당 발생하고 응답이 1~2초씩 걸린다. 매 요청마다 호출하면 사용자가 같은 화면을 다시 봐도 매번 돈과 시간을 쓴다. 그래서 먼저 `SPOT_RECOMMEND` 캐시를 확인하고, 캐시가 부족할 때만 Gemini를 호출하며, Gemini가 빈 결과나 예외를 던지면 `selectTrendingSpots`로 인기 여행지를 채운다. **세 단계는 각각 비용 절감 / 품질 / 가용성을 책임진다.** 면접에서 "캐시는 비용, 폴백은 가용성"이라고 한 줄로 정리하면 좋다.
:::

::: details 왜 마일리지를 30%로 막았나
마일리지는 보상으로 발급된 돈이라 100% 결제 가능하면 현금 매출 없이 상품이 빠져나간다. 그래서 `MAX_MILEAGE_RATE = 30`으로 항공권 금액의 30%만 마일리지로 쓰게 하고, 그 상한을 **천 원 단위 내림(floor)**으로 계산한다. 화면에서 보여주는 최대 사용액과 서버 검증 상한이 반올림 차이로 어긋나면 결제가 거부되므로, 양쪽 다 내림으로 맞춘 것이 디테일이다.
:::

## 5. 동작 원리 핵심 (표·작은 코드)

**추천 폴백 흐름**

```text
요청 → SPOT_RECOMMEND 캐시 조회
  ├─ 3건 이상  → 그대로 반환 (관심태그 재정렬)
  └─ 부족      → 최근 30건 로그 → 관심태그 추출 → 후보+tag_match_score
                 → Gemini 호출
                   ├─ 결과 있음 → FK 검증 후 SPOT_RECOMMEND 저장 → 반환
                   └─ 결과 없음/예외 → 트렌딩 폴백 반환
```

**항공권 혼합 결제 검증(추상화)**

```text
입력: cashAmount, mileageAmount
1) cashAmount + mileageAmount == finalPrice  (총액 일치)
2) mileageAmount <= floor(finalPrice * 30 / 100)  (마일리지 상한)
3) 보유 캐시/마일리지 >= 각 사용액  (잔액 충분)
→ 잔액 차감 + USER_WALLET_HISTORY 2건(CASH, MILEAGE) 기록
```

**자산·원장 구조**

| 항목 | 위치 | 역할 |
| --- | --- | --- |
| `cash_balance` / `mileage_balance` / `point_balance` | `USERS` | 현재 잔액 (빠른 조회용) |
| `USER_WALLET_HISTORY` | 별도 테이블 | 모든 증감 + 변동 후 잔액(balance_after) 원장 |
| `WALLET_REFUND_LOG` | 별도 테이블 | 환불 건 추적 |
| `SPOT_RECOMMEND` | 별도 테이블 | 추천 결과 캐시 |

## 6. 구현 상태 (정직하게)

면접에서 과장은 바로 들통난다. 됨/Mock/계획을 명확히 구분해 말하라.

| 기능 | 상태 |
| --- | --- |
| 개인화 추천 3단 폴백 (캐시·Gemini·트렌딩) | 구현됨 |
| 체류 로그·관심 태그·tag_match 정렬 | 구현됨 |
| Toss 캐시 충전 (prepare/complete/cancel) | 구현됨 (클라이언트 키 설정 시 활성) |
| 3원 지갑 + USER_WALLET_HISTORY 원장 | 구현됨 |
| 항공권 캐시+마일리지 혼합 결제·환불 | 구현됨 |
| 패키지 마켓플레이스 (SELLER 등록 → 관리자 승인) | 구현됨 (상태: DRAFT/PENDING/APPROVED/REJECTED) |
| **실제 외부 항공권 API 연동** | **Mock** (`MockFlightOfferProvider`, 실 API 미연동) |
| 추천 품질 정량 평가 체계 | 계획 (현재 없음, 향후 과제) |
| 모바일 반응형 / SPA | 계획 (JSP 데스크톱 레이아웃 위주) |

::: warning Mock을 숨기지 마라
"항공권은 외부 API를 붙였습니다"라고 말하면 안 된다. 정직한 표현은 **"외부 API 도입을 전제로 인터페이스를 설계해 두고, 현재는 Mock 프로바이더로 구매 플로우 전체를 구현했습니다"**다. 이게 오히려 설계 의도를 보여주는 강한 답변이다.
:::

## 7. 예상 꼬리질문 + 모범답안

**Q1. Gemini가 응답을 안 주거나 깨진 JSON을 주면요?**
A. 두 겹으로 막습니다. 파싱 단계에서는 코드펜스를 제거하고 `[` 위치부터 객체 단위로 중괄호 깊이를 세며 **완성된 객체만** 추출해, 응답이 토큰 한도로 잘려도 일부라도 살립니다. 그래도 결과가 비면 트렌딩 폴백으로 인기 여행지를 채워 화면이 비지 않게 합니다.

**Q2. 추천이 매번 똑같으면 사용자가 질리지 않나요?**
A. 프롬프트에 **이전 추천 spot_idx와 최근 방문 장소를 제외**하라고 명시하고, 후보 쿼리 레벨에서도 currentSpotIdx를 빼며, 추천끼리 서로 다른 지역·분위기를 갖도록 조건을 줍니다. 캐시도 무효화(invalidateCache)할 수 있어 반복을 줄였습니다.

**Q3. 왜 캐시·마일리지·포인트를 굳이 3개로 나눴나요? 하나면 단순한데.**
A. 회계와 정책이 다르기 때문입니다. 캐시는 사용자가 Toss로 충전한 실제 돈이라 환불 대상이고, 마일리지는 보상성 자산이라 사용처를 제한(예: 항공권 30%)하며, 포인트는 활동 보상으로 상점 등 다른 쓰임을 갖습니다. 한 잔액으로 합치면 환불 정산과 악용 방지가 불가능해집니다.

**Q4. 동시에 두 번 결제 요청이 오면 잔액이 음수가 될 수 있지 않나요?**
A. 결제 전 보유 잔액과 사용액을 비교해 거부하고, 차감과 원장 기록을 트랜잭션으로 묶습니다. 그리고 잔액 컬럼만 믿지 않고 `USER_WALLET_HISTORY`에 **balance_after**를 함께 남겨, 사후에 원장 합과 현재 잔액이 일치하는지 검증·정정할 수 있게 했습니다. (분산 환경의 완전한 동시성 제어는 향후 락 강화 과제로 봅니다.)

**Q5. 실제 항공 API를 붙이라고 하면 얼마나 걸리나요?**
A. 서비스·컨트롤러·화면·결제는 `FlightOfferProvider` 인터페이스에만 의존하므로, 그 인터페이스를 구현한 새 Provider 하나(예: Amadeus 어댑터)를 만들고 Mock 대신 주입하면 됩니다. 견적 가격이 외부에서 오므로 결제 검증의 상한 계산만 그대로 재사용됩니다. **변경 범위를 Provider 한 클래스로 가둔 것**이 이 추상화의 목적입니다.

**Q6. 추천이 추천한 spot_idx가 실제로 DB에 없으면요?**
A. 저장 직전에 후보 집합으로 FK를 검증해 유효하지 않은 spot_idx는 제외하고, 저장된 추천이 0건이면 트렌딩으로 폴백합니다. AI가 환각으로 없는 ID를 만들어도 DB 정합성이 깨지지 않습니다.
