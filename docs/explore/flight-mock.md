# 항공권 (Mock 프로바이더)

> 외부 항공 API 없이도 여행지마다 출발·귀국 왕복 견적을 보여주고, 실제 결제(캐시·마일리지)까지 도는 시뮬레이션. 핵심은 `FlightOfferProvider` 인터페이스 한 겹으로 데이터 소스를 갈아끼울 수 있게 만든 설계다.

## 1. 한 줄 정의

여행지 상세 화면에서 인천 출발 왕복 항공권 견적 목록을 보여주고 캐시·마일리지로 구매·취소까지 처리하는 기능이며, 현재 견적 데이터는 외부 API가 아니라 결정론적으로 생성되는 **Mock** 값이다.

## 2. 왜 이렇게 설계했나

실제 항공권 API(예: GDS, 항공사 제휴 API)는 계약·인증·비용·트래픽 제한이 따라온다. 학습·데모 단계에서는 그 의존성을 떠안기 어렵다. 그래서 두 가지를 분리했다.

- **데이터 소스 추상화:** 견적을 만드는 책임을 `FlightOfferProvider` 인터페이스로 빼고, 서비스·컨트롤러·JSP는 결과 DTO(`FlightOfferDto`)만 바라보게 했다. 나중에 Amadeus 같은 실제 프로바이더를 붙일 때 인터페이스 구현체 하나만 추가하면 윗단 코드가 거의 그대로 산다.
- **정직한 Mock:** 가짜 데이터지만 가격·항공편·소요시간이 여행지 좌표·이름에 따라 **그럴듯하게** 나오도록 규칙화했다. 같은 여행지·같은 날짜면 항상 같은 가격이 나오도록 결정론적으로 만들어, 화면을 새로고침해도 견적이 흔들리지 않는다.

핵심 의도는 "실제 결제·환불·지갑 차감 로직은 진짜로 돌리되, 외부 항공 API 의존성만 격리한다"는 것이다.

:::tip 추상화의 이득
교체 지점이 인터페이스 한 곳(`FlightOfferProvider`)으로 모여 있으면, 외부 연동 작업이 "전체 리팩터링"이 아니라 "구현체 추가"가 된다. 면접에서 강조하기 좋은 포인트다.
:::

## 3. 어떤 기술로 구현했나 (실제 클래스·테이블)

표준 4계층 위에 프로바이더 한 겹을 얹었다.

| 구성요소 | 실제 식별자 | 역할 |
| --- | --- | --- |
| 프로바이더 인터페이스 | `FlightOfferProvider` | 견적 조회 계약. `supports` / `getOffers` 정의, 최저가·단건 조회는 default 메서드 |
| Mock 구현체 | `MockFlightOfferProvider` (`@Component`) | 좌표·이름 기반 결정론적 가짜 견적 생성 |
| 견적 DTO | `FlightOfferDto` | 화면·결제가 바라보는 단일 모델 |
| 서비스 | `FlightServiceImpl` | 견적 조회, 회원 등급 할인, 결제·취소 트랜잭션 |
| 컨트롤러 | `FlightController` (`/flight`) | offers 조회 / purchase / cancel 엔드포인트 |
| 구매 기록 테이블 | `FLIGHT_PURCHASE_SIMULATION` | 구매 이력(purchase_no, used_cash, used_mileage, status 등) |
| 지갑 연동 | `WalletMapper` + `USER_PAYMENT_HISTORY` | 캐시·마일리지 차감, 결제·환불 이력 |

테이블 이름 자체가 `FLIGHT_PURCHASE_SIMULATION`이라는 점이 설계 의도를 그대로 드러낸다. 구매 행에는 `provider_type` 값으로 MOCK이 기록된다.

## 4. 동작 원리 (흐름·표·작은 코드)

### 견적이 만들어지는 규칙

`MockFlightOfferProvider`는 여행지(`ExploreVO`)의 좌표·이름·주소·설명 텍스트를 읽어 도착 공항과 가격대를 정한다.

- **국내 제외:** 위경도가 한국 범위(위도 33.0~39.6, 경도 124.0~132.0) 안이거나 이름에 한국 지명이 들어가면 `supports`가 false. 항공권 탭 자체를 띄우지 않는다.
- **도착 공항 매핑:** 텍스트에 도쿄가 있으면 NRT, 오사카·교토면 KIX, 방콕이면 BKK 같은 식으로 키워드 매핑. 못 찾으면 기본 DEST.
- **가격대:** 일본·중국·대만은 약 18만~45만, 동남아는 25만~65만, 유럽·미국은 90만 이상 등 권역별 범위.
- **결정론:** `spotIdx`·공항코드·출발일·귀국일을 해시한 seed로 기준가를 뽑는다. 같은 입력이면 같은 결과.
- **날짜 보정:** 주말 출발·귀국이면 +12%, 7일 이내 임박 출발이면 +18%, 45일 이상 여유면 -8%, 10일 이상 장기면 +6%.
- **3개 항공편:** 대형 항공사 2개 + 저비용 항공사 1개를 만들고 총액 오름차순으로 정렬해 반환한다. 가격은 천원 단위로 반올림.

```text
ExploreVO(좌표·이름) ─▶ supports? ─국내면─▶ 빈 목록
                              │
                            해외면
                              ▼
        도착공항 매핑 + 권역 가격대 + seed 해시 + 날짜 보정
                              ▼
        Korean Air / Asiana / 저비용 3건 → 총액 오름차순 정렬
```

### 견적 조회와 회원 할인

컨트롤러 `GET /flight/offers?spotIdx=...&departureDate=...&returnDate=...`가 진입점이다. `FlightServiceImpl`이 프로바이더에서 받은 원가(`totalPrice`)에 회원 등급 정책(`MEMBER_GRADE_POLICY`)의 할인율을 적용해 `finalPrice`를 채우고, 그 최종가 기준으로 마일리지 최대 사용액(총액의 30%, 천원 단위 내림)을 다시 계산한다. 로그인 안 한 사용자는 할인 0, 등급 BRONZE로 표시된다.

### 구매: 서버가 가격을 다시 검증

요청 본문에 클라이언트가 보낸 금액이 있어도 그대로 믿지 않는다. `purchase`는 `@Transactional` 안에서 다음을 강제한다.

| 검증 | 규칙 |
| --- | --- |
| 견적 재조회 | 서버가 같은 offerId로 Mock 견적을 다시 만들어 가격 산출 |
| 합계 일치 | cashAmount + mileageAmount 가 finalPrice 와 정확히 같아야 함 |
| 음수 금지 | 캐시·마일리지 모두 0 이상 |
| 마일리지 상한 | 마일리지는 최종가의 30%까지만 |
| 잔액 확인 | 행 잠금으로 사용자 조회 후 캐시·마일리지 잔액 검사 |

통과하면 지갑 잔액을 차감하고 `USER_PAYMENT_HISTORY`·지갑 이력·`FLIGHT_PURCHASE_SIMULATION`을 한 트랜잭션으로 기록한다. 구매번호는 FLT- 접두사 + UUID 16자리. 취소(`cancelPurchase`)는 status 가 COMPLETED 인 건만 허용하고, 사용한 캐시·마일리지를 환불 이력과 함께 되돌린다.

:::warning 동시성·정합성
잔액 차감은 행 잠금(for update) 조회 후 갱신하고, 결제 검증의 30% 상한과 화면에 표시하는 마일리지 최대 사용액 모두 같은 천원 단위 내림으로 맞춰 표시값이 검증 상한을 넘지 않게 했다. 이게 어긋나면 사용자는 화면대로 입력했는데 결제가 거절되는 모순이 생긴다.
:::

## 5. 구현 상태 (됨 vs Mock/계획)

| 항목 | 상태 |
| --- | --- |
| 권역별 견적 목록·출발/귀국 최저가 정렬 | 구현됨 (Mock 데이터) |
| 회원 등급 할인 적용 | 구현됨 |
| 캐시+마일리지 혼합 결제·서버 가격 검증 | 구현됨 |
| 구매 이력 저장·예매 취소·환불 | 구현됨 |
| 프로바이더 인터페이스로 소스 교체 가능 | 구현됨 (구조) |
| 실제 외부 항공 API 연동 | **미연동 (Mock 단계)** |
| 좌석 선택·실시간 잔여석·발권 | 계획/범위 외 |

정직하게: 결제·환불·지갑 정합성 같은 **백엔드 로직은 진짜로 동작**하고, 항공편·가격 **데이터만 가짜**다. 외부 API를 붙이려면 `FlightOfferProvider` 구현체를 추가하고 `provider_type` 값을 바꾸면 된다.

## 6. 면접 답변 3단계

1. **한 문장:** "항공권은 외부 API 없이도 권역별 왕복 견적을 보여주고 캐시·마일리지로 결제·취소까지 도는 시뮬레이션이며, 데이터 소스만 `FlightOfferProvider` 인터페이스로 격리해 둔 구조입니다."
2. **설계 의도:** "실제 결제·환불·잔액 정합성은 진짜로 돌리되 외부 항공 API 의존성만 분리하고 싶었습니다. 그래서 견적 생성 책임을 인터페이스로 빼고, 서비스·화면은 결과 DTO만 바라보게 했습니다. Mock도 좌표·이름 기반으로 결정론적이라 같은 여행지·날짜면 항상 같은 가격이 나옵니다."
3. **확장:** "Amadeus 같은 실제 프로바이더를 붙이면 인터페이스 구현체 하나만 추가하면 됩니다. 구매 행에 `provider_type`을 남겨 둬서 MOCK과 실연동을 구분·전환할 수 있게 했습니다."

## 7. 꼬리질문 + 모범답안

:::details 왜 가짜 데이터인데 결제 로직은 진짜로 돌리나요?
프로젝트의 학습 목표가 트랜잭션·지갑 정합성·할인·환불 처리였기 때문입니다. 외부 항공 API는 계약·비용·트래픽 제한이 커서 데모 단계에 떠안기 어렵습니다. 그래서 위험한 외부 의존성만 격리하고, 정작 검증하고 싶었던 결제 흐름은 실제로 구현해 검증 가능하게 두었습니다.
:::

:::details 클라이언트가 보낸 결제 금액을 그대로 쓰지 않는 이유는?
가격은 신뢰 경계 밖의 입력이기 때문입니다. `purchase`에서 서버가 같은 offerId로 견적을 다시 만들어 finalPrice를 산출하고, 캐시+마일리지 합계가 그 값과 정확히 일치하는지, 마일리지가 30% 상한을 넘지 않는지, 잔액이 충분한지를 모두 서버에서 검증합니다. 클라이언트 값을 믿으면 가격 조작이 가능합니다.
:::

:::details 같은 화면을 새로고침해도 가격이 같은 이유는?
`spotIdx`·도착공항코드·출발일·귀국일을 해시한 seed로 기준가를 뽑기 때문입니다. 랜덤이 아니라 입력의 함수라서 결정론적입니다. 덕분에 견적 조회와 구매 시점의 서버 재검증이 같은 값을 만들어 가격 불일치가 생기지 않습니다.
:::

:::details 실제 항공 API로 바꾸려면 무엇을 손대야 하나요?
`FlightOfferProvider`를 구현한 새 프로바이더를 만들고 Spring 빈으로 등록하면 됩니다. 컨트롤러·서비스의 결제 로직과 JSP는 `FlightOfferDto`만 바라보므로 거의 그대로 둘 수 있습니다. 다만 실제 API는 잔여석·발권·실시간 가격 변동이 있어 견적 만료, 발권 실패 보상 같은 처리를 추가해야 합니다.
:::

:::details 국내 여행지에는 왜 항공권이 안 보이나요?
`MockFlightOfferProvider.supports`가 여행지 좌표가 한국 범위 안이거나 이름에 한국 지명이 들어가면 false를 반환하기 때문입니다. 국내 여행은 항공권 판매 대상이 아니라는 제품 가정을 코드 수준에서 막아, 무의미한 인천발 국내 항공편이 노출되지 않게 했습니다.
:::

## 8. 직접 말해보기

- `FlightOfferProvider` 인터페이스가 어떤 의존성을 격리하는지, 그 덕에 실연동 전환이 왜 쉬워지는지 한 문장으로 말해보라.
- 결제 시 서버가 가격을 다시 검증하는 이유와, 검증에서 막는 위험(가격 조작·잔액 초과·마일리지 과다 사용)을 설명해보라.
- "Mock이라 의미 없는 기능 아닌가요"라는 질문에, 무엇이 진짜로 동작하고 무엇만 가짜인지 구분해 반박해보라.
