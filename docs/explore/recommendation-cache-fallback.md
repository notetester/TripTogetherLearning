# 추천 캐시·폴백

> AI 추천을 매번 LLM에 묻지 않는다. DB 캐시 → Gemini 호출 → 트렌딩 폴백의 3단 경로로 응답 속도와 비용을 동시에 잡는다.

허브: [도메인 전체 개요](/domains) · [담당별 보기](/by-area/) · [전체 흐름](/flow/)
이어 읽기: [AI 추천(Gemini)](/explore/ai-recommendation-gemini) · [탐색 필터](/explore/explore-filters) · [폴백 전략(AI 공통)](/ai/fallback-strategy)

## 1. 한 줄 정의

여행지 추천 응답을 만들 때, 5분 이내의 추천 결과가 DB에 남아 있으면 그대로 재사용하고, 없거나 부족하면 Gemini를 호출하며, AI가 비어 있으면 트렌딩 여행지로 떨어뜨리는 3단 폴백 파이프라인이다.

## 2. 왜 이렇게 설계했나

추천은 사용자가 여행지 상세를 볼 때마다 호출된다. 매 호출을 LLM에 직접 던지면 다음 세 가지가 동시에 나빠진다.

- 응답 지연: Gemini 왕복은 수백 ms에서 수 초까지 걸려 카드 영역이 비어 보인다.
- 비용: 외부 LLM 호출은 토큰 단위 과금이라 트래픽에 비례해 누적된다.
- 가용성: 외부 API가 실패하거나 응답이 잘려도 화면은 빈 칸이 되면 안 된다.

그래서 정책을 세 단계로 분리했다. 자주 바뀌지 않는 결과는 짧은 TTL 캐시로 흡수하고, 캐시가 없을 때만 LLM을 호출하며, LLM이 실패해도 항상 무언가는 보이도록 결정론적 폴백을 둔다. 즉 **빠른 경로는 DB만 읽고, 느리고 비싼 경로는 꼭 필요할 때만 탄다.**

:::tip 캐시 TTL을 5분으로 잡은 이유
추천은 사용자의 체류 이력에 반응해야 하므로 너무 길면 신선도가 떨어지고, 너무 짧으면 캐시 효과가 사라진다. 5분은 같은 세션 내 연속 탐색을 흡수하면서도 새 행동을 곧 반영할 수 있는 절충점이다. 게다가 사용자가 새 체류 로그를 남기면 즉시 캐시를 비워 다음 호출에서 신선한 추천을 다시 만든다.
:::

## 3. 어떤 기술로 구현했나(실제 클래스·테이블)

| 구성 요소 | 클래스 / 파일 | 역할 |
| --- | --- | --- |
| 진입점 | `RecommendController` | GET /recommend/spots, POST /recommend/view-log |
| 오케스트레이션 | `RecommendService.getRecommendations` | 3단 폴백 전체 흐름 |
| 데이터 접근 | `RecommendMapper` + recommendMapper.xml | 캐시 조회·후보·트렌딩·저장 |
| 외부 모델 | Gemini 2.5 Flash (REST, `RestTemplate`) | 후보 중 추천 3건 선택 |
| 후보 모델 | `CandidateSpotVO`, `RecommendVO`, `SpotViewLogVO` | 태그 매칭 점수·체류 로그 |

핵심 테이블은 네 개다.

- `SPOT_RECOMMEND`: 추천 결과 캐시. `created_at` 으로 TTL 판정.
- `SPOT_VIEW_LOG`: 체류 로그(stay_seconds). 관심 태그 추출·트렌딩 점수의 원천.
- `SPOT_TRAVEL`: 여행지 본체(후보 풀).
- `SPOT_TAG` / `SPOT_TAG_LIST`: 태그 매칭 점수 계산용.

LLM API 키는 `gemini.api.key` 프로퍼티로 주입되며, 문서에서는 자리표시자 `API_KEY` 로만 표기한다.

## 4. 동작 원리(흐름·표·작은 코드)

### 3단 경로 요약

| 단계 | 조건 | 행동 | 비용 |
| --- | --- | --- | --- |
| 1. 캐시 적중 | 5분 내 추천이 3건 이상 | DB 결과를 태그 정렬 후 즉시 반환 | DB 읽기만 |
| 2. Gemini 호출 | 캐시 없음 또는 부족 | 후보 추출 → Gemini로 3건 선택 → 저장 | LLM 왕복 |
| 3. 트렌딩 폴백 | 후보 없음 / AI 결과 없음 / 저장 0건 | 최근 7일 인기 여행지 3건 | DB 읽기만 |

```text
getRecommendations(userIdx, currentSpotIdx)
  ├─ ① 5분 캐시 조회 ── size >= 3 ? ──▶ 반환 (빠른 경로)
  │
  ├─ ② 체류 로그로 관심 태그 추출 → 후보 조회·재정렬
  │     candidates 비었나? ──────────▶ 트렌딩 폴백
  │     callGemini(...) → merge → ensureExactCount
  │     aiResults 비었나? ───────────▶ 트렌딩 폴백
  │     캐시 삭제 후 재저장 (FK 검증) → saved == 0 ? ─▶ 트렌딩 폴백
  │
  └─ ③ 저장본 재조회(태그 일치 정렬) → 상위 3건 반환
```

### 캐시 판정 SQL의 핵심

캐시 적중 여부는 시간 윈도우 하나로 결정된다.

```sql
SELECT ... FROM SPOT_RECOMMEND sr JOIN SPOT_TRAVEL st ...
WHERE sr.user_idx = #{userIdx}
  AND sr.created_at >= DATE_SUB(NOW(), INTERVAL 5 MINUTE)
ORDER BY tag_match_count DESC, sr.rec_score DESC
LIMIT 3
```

반환된 행이 3건 미만이면 캐시 미스로 보고 다음 단계로 내려간다. 빈 결과가 아니라 **정족수(3건)** 로 판정하는 점이 중요하다. 추천 카드는 항상 3장을 채워야 하므로, 2건만 캐시된 상태는 미스로 취급한다.

### 캐시 무효화(쓰기 경로)

읽기만 캐시를 다루는 게 아니다. 체류 로그가 쌓이면 `RecommendController.saveViewLog` 가 곧바로 `invalidateCache` 를 호출해 해당 사용자의 `SPOT_RECOMMEND` 행을 전부 지운다.

```java
recommendService.saveViewLog(userIdx, spotIdx, staySeconds);
recommendService.invalidateCache(userIdx); // 다음 호출에서 Gemini 재계산
```

이로써 캐시는 시간(5분 TTL)과 이벤트(새 행동) 두 축으로 만료된다. 사용자가 행동하면 신선도를, 행동이 없으면 절약을 택한다.

### 폴백 안에 한 겹 더: 후보 보강

Gemini 결과가 모자라더라도 곧장 트렌딩으로 떨어지지 않는다. `mergeWithTopCandidates` 와 `ensureExactRecommendationCount` 가 AI가 고른 항목에 태그 우선순위가 높은 후보를 채워 3건을 맞추고, 그래도 부족할 때만 트렌딩을 호출한다. 즉 폴백은 **개인화 후보 보강 → 트렌딩** 의 2겹 안전망이다.

### 트렌딩 폴백의 정의

최후 폴백은 결정론적이고 LLM과 무관하다.

```sql
ORDER BY
  (최근 7일 SPOT_VIEW_LOG 조회수) DESC,
  (SPOT_LIKE 좋아요 수) DESC
LIMIT 3
```

컨트롤러는 첫 항목의 rec_idx 가 0인지로 트렌딩 여부를 판별해 응답에 isTrending 플래그를 실어 보낸다. 프런트는 이 플래그로 요즘 뜨는 여행지 같은 라벨을 다르게 노출할 수 있다.

## 5. 구현 상태(됨 vs Mock/계획)

| 항목 | 상태 |
| --- | --- |
| 5분 TTL 캐시(SPOT_RECOMMEND) | 구현됨 |
| 체류 로그 기반 캐시 무효화 | 구현됨 |
| Gemini 2.5 Flash 호출·잘린 JSON 복구 파싱 | 구현됨 |
| 후보 보강 + 트렌딩 2겹 폴백 | 구현됨 |
| 트렌딩(7일 조회 + 좋아요) 폴백 | 구현됨 |
| 캐시 저장소 | DB(SPOT_RECOMMEND) 한 곳. Redis 등 인메모리 계층 없음 |
| 캐시 적중률·폴백 비율 지표화 | 미구현(로그만 존재, 정량 모니터링은 향후 과제) |
| AI 추천 품질 정량 평가 | 미구현(프로젝트 공통 과제) |

:::warning 솔직한 한계
캐시는 RDB 한 곳에만 있어, 매 추천 호출이 `SPOT_RECOMMEND` 읽기를 동반한다. 트래픽이 커지면 인메모리 캐시 한 겹을 앞에 두는 게 자연스러운 다음 단계다. 또한 5분 TTL과 정족수 3건은 코드 상수로 고정되어 있어 런타임 튜닝이 불가능하다.
:::

## 6. 면접 답변 3단계

1. 한 문장: 여행지 추천을 DB 캐시 조회 → Gemini 호출 → 트렌딩 폴백의 3단으로 짜서, 빠른 경로는 DB만 읽고 비싼 LLM은 꼭 필요할 때만 타도록 응답 속도와 비용을 최적화했습니다.
2. 한 단계 더: 캐시는 5분 TTL이고, 사용자가 새 체류 로그를 남기면 즉시 무효화해 신선도를 보장합니다. AI가 실패하거나 결과가 비어도 개인화 후보 보강과 트렌딩 폴백으로 항상 3건을 채웁니다.
3. 트레이드오프: 캐시 신선도와 절약 사이는 TTL 길이로, 가용성은 결정론적 폴백으로 맞췄습니다. 다만 캐시가 RDB 한 곳뿐이라 트래픽이 커지면 인메모리 계층을 앞단에 두는 게 다음 개선점입니다.

## 7. 꼬리질문+모범답안

:::details 캐시 적중을 빈 결과가 아니라 3건 이상으로 판정하는 이유는?
추천 UI는 항상 카드 3장을 채워야 합니다. 캐시에 2건만 있으면 화면이 부족하므로 미스로 보고 재계산해 3건을 확보합니다. 정족수 기준이 빈 칸 노출을 막습니다.
:::

:::details 캐시가 만료되지 않았는데도 추천이 갱신돼야 하는 경우는?
사용자가 새 여행지에 머물러 체류 로그를 남기면 관심 프로필이 바뀝니다. 이때 view-log 저장 직후 invalidateCache로 캐시를 비워, 다음 호출에서 Gemini가 최신 행동을 반영해 다시 계산하도록 했습니다. 시간 만료와 이벤트 만료를 함께 씁니다.
:::

:::details Gemini가 응답을 반환했는데 추천이 3건이 안 되면?
먼저 태그 우선순위가 높은 미방문 후보로 채워 3건을 맞춥니다. 그래도 부족하면 최근 7일 조회수와 좋아요 기준의 트렌딩으로 떨어집니다. AI 결과 부족이 곧장 트렌딩으로 직행하지 않고 개인화를 한 겹 더 유지합니다.
:::

:::details Gemini가 후보에 없는 spot_idx를 추천하면 어떻게 막나요?
저장 직전에 후보 집합(validIdx)에 포함된 spot_idx 만 통과시킵니다. 후보 밖 값은 로그를 남기고 버려 존재하지 않는 여행지나 FK 위반을 차단합니다. 외부 모델 출력을 신뢰하지 않고 화이트리스트로 검증하는 패턴입니다.
:::

:::details 응답이 잘린 JSON으로 와도 동작하나요?
parseGeminiResponse가 코드펜스를 제거하고 배열 시작 위치부터 중괄호 깊이를 세며 완성된 객체만 추출합니다. 마지막 객체가 잘려 있으면 그 항목만 버리고 앞의 완성분을 살립니다. 부분 응답에서도 가능한 만큼 회수하는 방어적 파싱입니다.
:::
