# Git 협업 전략

> 4명이 한 저장소를 나눠 개발하면서 충돌을 줄인 핵심은 "개인 브랜치 + dev로 PR + 머지 커밋 보존"이라는 단순한 규칙 세 개였다.

## 1. 한 줄 정의

TripTogether는 **개인 브랜치에서 작업하고, `dev` 브랜치로 Pull Request를 올려 머지하는 GitHub Flow 변형**을 썼다. 보호 브랜치(`dev`/`main`/`master`/`live`)에는 직접 push하지 않고, 모든 변경은 PR을 거쳐 들어간다.

::: tip 숫자로 본 규모
약 858개 커밋, 그중 **155개가 머지 커밋**(약 108개의 GitHub PR 머지 + 약 47개의 `dev → 개인 브랜치` 동기화 머지)으로 구성된 4인 공동 개발 저장소다. 즉, 직접 작성한 변경 커밋이 700개 안팎, 나머지가 통합 흔적이다.
:::

## 2. 왜 이렇게 설계했나

도메인을 **수직 분담**(auth, community, courses, explore/commerce, inquiry/notification, admin/superAdmin, AI, i18n 등 약 14~15개 모듈)했기 때문에, 협업 전략도 그 분담을 깨지 않게 설계했다.

- **개인 브랜치 격리** — 각자 자기 도메인을 자기 브랜치에서 끝까지 만들고, 완성 단위로만 `dev`에 합친다. 미완성 코드가 다른 사람 작업을 막지 않는다.
- **PR을 통합 게이트로** — 모든 변경이 PR 하나를 거치므로, 변경 범위가 한눈에 보이고 코드 리뷰·되돌리기 단위가 PR로 정렬된다. `git revert <merge>`로 PR 단위 롤백이 가능하다.
- **보호 브랜치 직접 push 금지** — `dev`가 항상 "통합되었고 빌드되는 상태"를 유지한다. 누구도 로컬에서 바로 `dev`를 밀지 않으므로, 깨진 커밋이 공유 브랜치에 들어갈 확률이 낮다.
- **머지 커밋 보존(non-fast-forward)** — squash·rebase로 히스토리를 평탄화하지 않고 머지 커밋을 남긴다. "이 PR이 언제, 어떤 브랜치에서 통합됐는가"가 그래프에 그대로 남아 추적성이 좋다.

:::details squash/rebase 대신 merge를 택한 트레이드오프
머지 커밋 방식은 히스토리가 다소 지저분해 보이는 단점이 있지만, (1) PR 단위 되돌리기가 쉽고 (2) 통합 시점이 명확하며 (3) 여러 사람이 같은 시기에 작업할 때 누구의 PR이 먼저 들어갔는지 시간순으로 읽힌다. 4인 동시 개발에서는 "깔끔한 직선 히스토리"보다 "추적 가능한 통합 기록"이 실용적이라는 판단이다.
:::

## 3. 어떤 기술로 구현했나 (실제 브랜치·규칙)

| 구성 요소 | 실제 값 / 규칙 |
| --- | --- |
| 기본 통합 브랜치 | `origin/HEAD → origin/dev` (저장소 기본 브랜치가 `dev`) |
| 보호 브랜치 | `dev`, `main`, `master`, `live` — 직접 push 금지 |
| 개발 브랜치 | 개발자별 개인 브랜치(예: `이름-기반 개인 브랜치`), 필요 시 `-TEMP`/`-TEMP2` 실험 브랜치 |
| 통합 방식 | GitHub Pull Request → `dev`로 **Merge commit** |
| 역방향 동기화 | `Merge branch 'dev' into <개인브랜치>`로 최신 `dev`를 주기적으로 끌어옴 |
| 커밋 메시지 | **한국어 본문**, `feat:`/`fix:`/`docs:`/`chore:`/`refactor:` prefix(영어 유지) |
| 릴리스 채널(추정) | `dev`(통합) → `main`/`master`(안정) → `live`(운영) 계열로 분리 |

커밋 메시지는 실제로 한국어 본문으로 작성됐다. 예: `explore 그리드 깨짐 수정`, `login-risk 7개 페이지 자체 toolbar·컬럼폭·정렬 인디케이터 페이지별 개별 최적화`. prefix는 영어, 설명은 한국어라는 규칙이 일관되게 지켜졌다.

::: warning 커밋/PR에 AI 도구 흔적 금지
공개 저장소 정책상 커밋 메시지와 PR 본문에 AI 도구(예: 코드 어시스턴트) 서명·`Co-Authored-By` 트레일러·"Generated with ..." 문구·이모지 서명을 넣지 않는다. PR 본문에는 **변경 사항만** 사실 위주로 적는다.
:::

## 4. 동작 원리 (흐름·표·작은 명령)

### 기능 한 건이 dev에 들어가기까지

```text
개인 브랜치 분기  →  커밋(한국어 메시지)  →  최신 dev 동기화  →  PR 생성
        →  (셀프/팀) 리뷰  →  Merge commit으로 dev에 통합  →  배포 채널로 승격
```

### 일상 작업 순서 (명령 추상화)

```bash
# 1) 최신 dev 기준으로 시작
git checkout dev && git pull
git checkout <개인-브랜치> && git merge dev   # dev를 개인 브랜치로 끌어와 동기화

# 2) 자기 도메인 작업 후 커밋 (한국어 본문, 영어 prefix)
git commit -m "feat: 커뮤니티 태그 공출현 카운트 캐시 추가"

# 3) push 후 GitHub에서 dev로 PR
git push origin <개인-브랜치>
#   → PR 생성, 리뷰, Merge(머지 커밋) 으로 dev 통합
```

### push 전 점검 루틴

push 요청 시 (1) 현재 브랜치가 보호 브랜치인지 확인하고 — 맞으면 멈춘다, (2) `git fetch origin`으로 `origin/dev`에 새 커밋이 있는지 보고 내가 만진 파일과 겹치는지 분석한 뒤, (3) "그대로 push / 머지 후 push"를 정하고, (4) 커밋 메시지에 AI 흔적이 없는지 확인한 다음 push한다. 이 순서가 머지 충돌을 push 시점이 아니라 push **이전**으로 당겨준다.

### 머지 충돌이 생기는 지점과 해소

수직 분담이라 도메인 코드 충돌은 드물지만, **공통 영역**에서 부딪힌다.

| 충돌 다발 지점 | 이유 | 완화책 |
| --- | --- | --- |
| `*.sql`(스키마) | 모두가 테이블을 추가/수정 | 스키마 변경은 소유자(팀장) 합의 후 반영 |
| `messages*.properties`(i18n) | 4개국어 키를 여러 도메인이 추가 | 키 네임스페이스를 도메인 prefix로 분리 |
| 공통 인터셉터·라우팅·공통 컴포넌트 | 체인/전역 설정 한 파일 집중 | 변경 전 팀 합의, 작은 PR로 자주 통합 |
| 빌드 설정(`pom.xml`) | 의존성 추가 동시 발생 | 의존성 변경은 별도 PR로 분리 |

충돌이 났을 때는 개인 브랜치에서 `git merge dev`로 먼저 해소하고, 충돌 없는 깨끗한 상태를 push해 PR을 갱신한다. `dev`에서 충돌을 푸는 것이 아니라 **내 브랜치로 가져와 푸는** 방향이 핵심이다(보호 브랜치를 깨끗하게 유지).

## 5. 구현 상태 (됨 vs 계획)

| 항목 | 상태 |
| --- | --- |
| 개인 브랜치 + dev PR 머지 워크플로우 | ✅ 실제 운용 (155 머지 커밋, ~108 PR) |
| 한국어 커밋 메시지 + 영어 prefix 규칙 | ✅ 히스토리에서 일관 확인 |
| 보호 브랜치 분리(`dev`/`main`/`master`/`live`) | ✅ 원격에 존재 |
| `dev → 개인 브랜치` 주기적 동기화 머지 | ✅ 다수 존재 |
| PR 단위 코드 리뷰 / 머지 충돌 해소 문화 | ✅ PR 기반 통합으로 운용 |
| CI 자동 빌드·테스트 게이트(예: GitHub Actions) | ⚠️ 저장소 근거상 미확인 — 자동 게이트는 계획/추정 영역 |
| 브랜치 보호 규칙의 강제(필수 리뷰어·상태 체크) | ⚠️ 규칙은 합의 기반, 도구 강제 여부는 미확인 |

:::warning 정직한 한계
저장소 히스토리로 확인되는 것은 "PR 머지로 dev에 통합했다"는 사실까지다. 자동 CI 파이프라인, 필수 리뷰어 강제, 코드 커버리지 게이트 등은 근거가 없어 **확정해 말하지 않는다**. 면접에서도 "PR 기반 통합과 머지 커밋 보존"까지만 단정하고, CI는 "도입했다면 빌드·테스트 게이트를 붙였을 것"이라고 가정형으로 말하는 편이 안전하다.
:::

## 6. 면접 답변 3단계

1. **한 문장** — "4명이 도메인을 수직 분담해서, 각자 개인 브랜치에서 작업하고 `dev`로 PR을 올려 머지하는 GitHub Flow 변형을 썼습니다."
2. **근거 한 겹** — "보호 브랜치엔 직접 push하지 않고 PR을 통합 게이트로 뒀습니다. squash 대신 머지 커밋을 남겨서 PR 단위로 추적·되돌리기가 가능하게 했고, 충돌이 잦은 스키마·i18n·공통 설정은 변경 전 합의하는 규칙을 뒀습니다."
3. **트레이드오프·수치** — "커밋 약 858개 중 155개가 머지 커밋입니다. 깔끔한 직선 히스토리는 포기했지만, 4인 동시 개발에선 통합 시점 추적성과 PR 단위 롤백이 더 중요하다고 판단했습니다."

## 7. 꼬리질문 + 모범답안

:::details Q1. 왜 squash merge가 아니라 merge commit을 썼나요?
PR 단위 되돌리기(`git revert <merge>`)와 통합 시점 추적이 쉽기 때문입니다. 4명이 같은 시기에 작업해서, "어떤 PR이 언제 dev에 들어갔는가"를 그래프로 읽는 가치가 직선 히스토리의 깔끔함보다 컸습니다. 다만 히스토리가 다소 복잡해지는 비용은 인정합니다 — 정리가 중요한 팀이라면 squash로 바꿀 수 있는 트레이드오프입니다.
:::

:::details Q2. 머지 충돌은 주로 어디서 났고 어떻게 줄였나요?
도메인 코드는 수직 분담이라 충돌이 드물었고, 충돌은 공통 파일(스키마 `.sql`, 4개국어 `messages.properties`, 공통 인터셉터·라우팅, `pom.xml`)에 집중됐습니다. i18n 키를 도메인 prefix로 네임스페이스 분리하고, 스키마·공통 영역 변경은 소유자 합의 후 작은 PR로 자주 통합해 충돌 범위를 좁혔습니다. 해소는 항상 개인 브랜치에서 `git merge dev`로 먼저 풀고 깨끗한 상태를 push했습니다.
:::

:::details Q3. dev에 깨진 코드가 들어가는 걸 어떻게 막았나요?
보호 브랜치 직접 push를 금지하고 모든 변경을 PR로 받았습니다. push 전 `git fetch`로 origin/dev 변경과 내 파일의 겹침을 먼저 확인하는 루틴을 둬서, 충돌·회귀를 push 이전에 잡았습니다. 자동 CI 게이트까지 있었다면 더 강했겠지만, 그 부분은 저장소 근거가 없어 단정하지 않겠습니다.
:::

:::details Q4. 브랜치에 -TEMP 같은 게 있던데 무슨 용도인가요?
개인 브랜치(`-TEMP`/`-TEMP2`)는 큰 리팩터링이나 위험한 실험을 본 작업 브랜치와 분리하려는 임시 브랜치입니다. 실험이 성공하면 본 브랜치로 병합해 PR을 올리고, 실패하면 버립니다. 본 브랜치의 PR 흐름을 더럽히지 않으려는 격리 장치입니다.
:::

:::details Q5. 커밋 메시지 규칙이 따로 있었나요?
`feat:`/`fix:`/`docs:`/`chore:`/`refactor:` prefix는 영어로 유지하고, 본문은 한국어로 작성했습니다(예: `fix: explore 그리드 깨짐 수정`). 그리고 공개 저장소라 커밋·PR 본문에 AI 도구 흔적(서명·트레일러·이모지)을 넣지 않는 규칙을 강제했습니다. PR 본문은 변경 사항만 사실 위주로 적었습니다.
:::

## 8. 직접 말해보기

아래를 막힘 없이 30초 안에 말할 수 있으면 충분하다.

1. "우리 Git 전략을 한 문장으로?" → 개인 브랜치 + dev PR 머지 + 보호 브랜치 직접 push 금지.
2. "왜 머지 커밋을 남겼나?" → PR 단위 추적·롤백, 4인 동시 개발의 통합 시점 가시성.
3. "충돌은 어디서, 어떻게?" → 공통 파일(스키마·i18n·공통 설정)에서. 도메인 prefix 분리 + 개인 브랜치에서 `merge dev`로 선해소.
4. "정직한 한계는?" → 자동 CI 게이트·필수 리뷰어 강제는 근거 없어 단정하지 않음.
