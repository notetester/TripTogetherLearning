# AI 답변 초안 (Claude)

> 운영자가 버튼 한 번으로 Claude Haiku 답변 초안을 받아 검토·보완 후 등록한다. AI는 사람을 대체하지 않고 응대 시간을 줄이는 보조 도구다.

## 1. 한 줄 정의

문의 상세 화면에서 운영자가 AI 초안 버튼을 누르면, 해당 문의의 카테고리·제목·본문을 Claude Haiku에 단발 호출해 한국어 답변 초안을 생성하고, 답변 작성 textarea에 채워주는 운영자 전용 보조 기능이다.

## 2. 왜 이렇게 설계했나

- **초안은 보조, 등록은 사람.** AI가 만든 텍스트가 바로 사용자에게 나가지 않는다. 초안은 운영자 입력창에 채워질 뿐이고, 실제 답변 등록은 별도 버튼(`/answer`)으로 사람이 확정한다. 책임 소재와 품질 통제를 운영자가 쥔다.
- **실패해도 업무가 멈추면 안 된다.** AI 호출이 실패하면 빈 문자열을 반환하는 fail-safe 구조다. API 장애가 곧 답변 불가로 이어지지 않고, 운영자는 평소처럼 직접 작성하면 된다.
- **비용·속도 우선.** 문의 답변 초안은 짧고 정형적이라 대형 모델이 필요 없다. 속도와 비용이 유리한 Haiku 계열을 골라 운영자가 클릭 후 기다리는 시간을 줄였다.
- **권한은 운영자에 한정.** 사용자에게 노출되는 기능이 아니다. 컨트롤러에서 운영자 여부를 먼저 확인하고, 아니면 403으로 막는다.
- **싱글턴 호출로 단순화.** 사이트 도우미 같은 멀티턴 대화가 아니라 문의 한 건당 한 번 호출이다. 대화 히스토리를 저장·관리할 필요가 없어 구조가 단순하다.

## 3. 어떤 기술로 구현했나 (실제 클래스·테이블)

| 구성 | 실제 이름 |
| --- | --- |
| 서비스 | `InquiryAiService.generateDraft(category, title, content)` |
| 컨트롤러 | `InquiryController#aiDraft` — `POST /inquiry/{inquiryId}/ai-draft` |
| 모델 | Claude Haiku (`claude-haiku-4-5-20251001`) |
| API | Anthropic Messages API (`/v1/messages`, 버전 헤더 2023-06-01) |
| HTTP | `RestTemplate` (헤더 x-api-key, anthropic-version) |
| 입력 데이터 | `INQUIRY_POST` (category / title / content) |
| 답변 저장 | `INQUIRY_ANSWER` (등록 시점에 별도 흐름으로 저장) |
| 화면 | `views/inquiry/detail.jsp` — 버튼 id aiDraftBtn |

핵심은 입력과 출력의 분리다. AI 초안 기능은 `INQUIRY_POST`를 **읽기만** 한다. 초안 텍스트를 DB에 저장하지 않고 화면 textarea(`adminContent`)에만 채운다. 실제 `INQUIRY_ANSWER` 행은 운영자가 답변 등록을 눌렀을 때 생긴다.

```java
// InquiryAiService 핵심 (추상화)
String userMessage = "문의 유형: " + categoryLabel + "\n"
                   + "제목: " + title + "\n"
                   + "내용: " + content;

Map<String, Object> body = Map.of(
    "model",      "claude-haiku-4-5-20251001",
    "max_tokens", 1024,
    "system",     SYSTEM_PROMPT,                       // CS 담당자 역할·한국어·초안만 출력
    "messages",   List.of(Map.of("role", "user", "content", userMessage))
);
// 호출 실패·파싱 오류 시 "" 반환 (fail-safe)
```

시스템 프롬프트는 AI를 CS 담당자 역할로 고정하고, 인사말로 시작해 마무리 인사로 끝내며, 초안 텍스트만 출력하도록 제약한다. 카테고리 코드(service / payment / account / bug / etc)는 한국어 레이블로 매핑해 프롬프트에 넣는다.

## 4. 동작 원리 (흐름·표·작은 코드)

```text
[운영자] AI 초안 버튼 클릭
   │  fetch POST /inquiry/{id}/ai-draft
   ▼
[Controller] 운영자 여부 확인 ── 아니면 403
   │  문의 조회 ── 없으면 404
   ▼
[InquiryAiService] category+title+content → 프롬프트 조립
   │  Claude Haiku 단발 호출 (max_tokens 1024)
   ▼
   성공 → 초안 텍스트 / 실패 → "" (fail-safe)
   │
   ▼
[Controller] 초안 비어 있으면 500, 아니면 { success:true, draft }
   │
   ▼
[화면] textarea(adminContent)에 초안 주입 → 운영자 검토·수정
   │
   ▼
[운영자] 답변 등록 버튼 → POST /inquiry/{id}/answer (별도 호출)
```

응답 형태와 처리:

| 상황 | HTTP | 응답/동작 |
| --- | --- | --- |
| 운영자 아님 | 403 | 메시지(국제화 키), 초안 미생성 |
| 문의 없음 | 404 | success=false |
| 초안 비어 있음(AI 실패) | 500 | aiDraftFailed 메시지 |
| 성공 | 200 | draft 텍스트 반환 → textarea 채움 |

프런트는 단순하다. 버튼을 누르면 로딩 표시 후 fetch, 성공하면 초안을 입력창에 넣고, 실패하면 알림만 띄운다.

```js
const res  = await fetch(ctx + '/inquiry/' + inquiryId + '/ai-draft', { method: 'POST' });
const data = await res.json();
if (data.success) document.getElementById('adminContent').value = data.draft;
else alert(data.message || inquiryMessages.aiDraftFail);
```

## 5. 구현 상태 (됨 vs Mock/계획)

:::tip 구현된 것
- 운영자 전용 `POST /inquiry/{id}/ai-draft` 엔드포인트와 권한 가드(403)
- Claude Haiku 단발 호출, 카테고리·제목·본문 기반 한국어 초안 생성
- 실패 시 빈 문자열 반환 fail-safe, 빈 초안이면 500 응답
- 초안을 답변 textarea에 주입 → 검토 후 별도 등록(`/answer`)으로 확정
- 답변/초안 관련 UI 문구의 다국어(국제화 메시지 키) 처리
:::

:::warning Mock·계획·한계
- **초안 품질 정량 평가 체계 없음.** AI 응답 품질을 수치로 측정·회귀 검증하는 하니스는 아직 없다(프로젝트 공통 향후 과제).
- **초안 자체는 저장하지 않음.** 운영자가 등록하지 않으면 초안은 화면을 떠나는 순간 사라진다. 초안 채택률·재생성 횟수 같은 운영 지표는 수집하지 않는다.
- **단발 호출 한정.** 운영자가 "더 짧게/공손하게" 같은 후속 지시로 다듬는 멀티턴은 없다. 매 클릭이 독립 호출이다.
:::

## 6. 면접 답변 3단계

1. **한 문장.** "문의 상세에서 운영자가 버튼을 누르면 Claude Haiku가 카테고리·제목·본문을 보고 한국어 답변 초안을 만들어 입력창에 채워주고, 운영자가 검토·수정 후 등록하는 보조 기능입니다."
2. **설계 의도.** "AI를 의사결정자가 아니라 보조 도구로 뒀습니다. 초안은 DB에 저장하지 않고 화면에만 채우며, 실제 등록은 사람이 별도 버튼으로 확정합니다. 호출이 실패해도 빈 문자열을 돌려 답변 업무가 멈추지 않게 했습니다."
3. **트레이드오프.** "단발 호출이라 멀티턴 다듬기는 없고, 속도·비용을 위해 Haiku를 골랐습니다. 대신 초안 품질을 정량 평가하는 체계는 아직 없어 향후 과제로 남겨뒀습니다."

## 7. 꼬리질문 + 모범답안

:::details AI 초안이 부적절하면 사용자에게 그대로 나가나요?
아니요. 초안은 운영자 입력창에만 채워지고 DB에 저장되지 않습니다. 실제 답변 등록은 별도 엔드포인트(`/answer`)를 사람이 눌러야 발생합니다. AI 출력과 사용자 노출 사이에 항상 운영자 검토 단계가 있습니다.
:::

:::details API 호출이 실패하면 어떻게 되나요?
서비스가 예외를 잡아 빈 문자열을 반환합니다(fail-safe). 컨트롤러는 초안이 비어 있으면 500과 실패 메시지를 주고, 운영자는 평소처럼 직접 답변을 작성합니다. AI 장애가 답변 불가로 번지지 않는 구조입니다.
:::

:::details 왜 멀티턴이 아니라 단발 호출인가요?
문의 답변 초안은 한 건당 한 번 생성하면 충분한 작업입니다. 대화 맥락을 이어갈 필요가 없어 히스토리 저장·관리 비용을 없앴고, 구조가 단순해집니다. 멀티턴이 필요한 사이트 도우미 같은 기능과는 의도적으로 분리되어 있습니다.
:::

:::details 왜 다른 큰 모델이 아니라 Haiku인가요?
초안은 짧고 정형적이라 대형 모델의 추론력이 필요하지 않습니다. 운영자가 클릭 후 기다리는 체감 시간을 줄이는 게 더 중요해서, 속도와 비용이 유리한 Haiku 계열을 골랐습니다. 품질이 부족하면 운영자가 바로 수정합니다.
:::

:::details 권한 통제는 어디서 하나요?
컨트롤러에서 운영자 여부를 먼저 확인하고, 아니면 403으로 차단합니다. 사용자에게는 버튼 자체가 노출되지 않습니다. 문의가 존재하지 않으면 404로 응답합니다.
:::

## 8. 직접 말해보기

- 이 기능에서 AI 출력과 사용자 노출 사이에 사람이 끼는 지점은 정확히 어디인가? 한 문장으로 말해보라.
- 모델 호출이 실패했을 때 사용자·운영자 각각의 경험이 어떻게 달라지는지 설명해보라.
- "초안을 DB에 저장하지 않는다"는 선택의 장점과, 그로 인해 포기한 운영 데이터를 한 가지씩 말해보라.
