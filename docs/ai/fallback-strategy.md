# 폴백 전략

> 외부 LLM은 느려지고 끊기고 가끔 헛소리를 한다. TripTogether는 "AI가 실패해도 화면은 항상 무언가를 보여준다"를 원칙으로, 추천은 3단 폴백, 챗봇은 fast-path와 부적절 즉답으로 호출 자체를 줄이고, 마지막엔 안전 응답으로 떨어진다.

## 1. 한 줄 정의

폴백 전략은 **AI 호출이 느리거나 실패하거나 비정상 응답을 줄 때 사용자에게 빈 화면이나 에러 대신 품질이 한 단계 낮더라도 유효한 결과를 보장하는 다단계 안전망**이다. TripTogether에서는 (1) 추천의 `캐시 → Gemini → 트렌딩` 3단 폴백, (2) 챗봇의 fast-path·부적절 즉답으로 LLM 호출을 건너뛰는 절감 경로, (3) 모든 AI 호출 실패 시 미리 만든 안전 응답으로 수렴하는 방식으로 구현되어 있다.

## 2. 왜 이렇게 설계했나

여행 플랫폼에서 AI는 핵심이지만 **단일 장애점이 되면 안 된다.** 외부 모델 API는 우리 통제 밖이라 5xx, 타임아웃, 쿼터 초과, 안전 필터 차단, 깨진 JSON이 언제든 일어난다.

- **가용성**: 추천 위젯이나 챗봇 답변이 비어 있으면 사용자는 사이트 자체가 고장 났다고 느낀다. AI가 죽어도 트렌딩 여행지나 정적 안내 링크는 나와야 한다.
- **비용·토큰 절감**: "커뮤니티 어디야" 같은 단순 네비게이션이나 욕설 메시지에까지 본 LLM을 호출하면 토큰과 등급 쿼터가 낭비된다. LLM 없이 답할 수 있는 건 LLM 없이 답한다.
- **지연**: fast-path는 외부 왕복(수백 ms~수 초)을 0에 가깝게 줄여 체감 속도를 높인다.
- **신뢰성**: LLM이 고른 `spot_idx`나 만든 URL을 그대로 믿지 않는다. 검증을 통과하지 못한 항목을 버리고도 결과 개수를 맞춰야 하므로, 폴백은 "에러 처리"가 아니라 정상 흐름의 일부로 들어간다.

:::tip 핵심 멘탈 모델
폴백은 try-catch 한 줄이 아니라 **계층**이다. 위 계층일수록 빠르고 싸고 개인화되어 있으며, 아래로 내려갈수록 일반적이지만 항상 동작한다. 설계 목표는 "가능한 한 위 계층에서 끝내되, 끝까지 못 가도 최하단은 절대 실패하지 않게" 하는 것이다.
:::

## 3. 어떤 기술로 구현했나 (실제 클래스·테이블)

| 영역 | 클래스 / 테이블 | 폴백 역할 |
| --- | --- | --- |
| 추천 3단 폴백 | `RecommendService` | 캐시 적중 → Gemini 호출 → 트렌딩 순으로 하강 |
| 추천 후보·트렌딩 조회 | `RecommendMapper` (selectRecommendSpots, selectTrendingSpots) | 캐시 조회와 최종 트렌딩 쿼리 |
| 챗봇 오케스트레이션 | `ChatbotService` | fast-path·부적절 즉답·LLM 실패 폴백 분기 |
| 빠른 경로 | `ChatbotFastPathService` | 15자 이하 단순 요청을 LLM 없이 즉답 |
| 1차 분류 | `IntentContextService` | 부적절 사전 판정 + Gemini 실패 시 규칙 기반 분류 |
| 응답 VO | `ChatbotResponseVO`, `RecommendVO` | message/links/inappropriate, 추천 결과 |

추천이 사용하는 테이블은 체류 로그 `SPOT_VIEW_LOG`, 추천 캐시 `SPOT_RECOMMEND`이고, 트렌딩은 인기 지표 기반 쿼리(`selectTrendingSpots`)로 후보를 뽑는다. 챗봇 쪽 등급 쿼터는 `CHATBOT_GRADE_QUOTA`, 사용량은 일일 집계 테이블에 기록된다.

상세한 모델별 구현은 [Gemini 챗봇·추천](/ai/gemini-chatbot), [다중 AI 모델 통합](/ai/multi-model), [구조화 출력](/ai/structured-outputs)에서 다룬다.

## 4. 동작 원리 (흐름·표·작은 코드)

### 4-1. 추천 3단 폴백 (RecommendService)

`getRecommendations(userIdx, currentSpotIdx)`는 위에서 아래로 하강하며, 어느 단계든 3건을 채우면 즉시 반환한다.

| 단계 | 조건 | 동작 | 비용 |
| --- | --- | --- | --- |
| 1. 캐시 | `SPOT_RECOMMEND`에 3건 이상 | 관심 태그로 재정렬해 반환 | LLM 호출 0 |
| 2. Gemini | 캐시 미달 | 후보 + 관심 태그로 LLM 호출, id 검증 후 저장 | LLM 1회 |
| 3. 트렌딩 | 후보 없음·LLM 결과 없음·저장 0건 | `selectTrendingSpots`로 인기 여행지 반환 | LLM 호출 0 |

핵심은 트렌딩으로 떨어지는 분기가 **여러 군데**라는 점이다. 후보 집합 자체가 비었을 때, Gemini가 빈 배열을 줬을 때, 검증을 통과해 실제로 저장된 결과가 0건일 때 모두 같은 안전 응답으로 수렴한다.

```java
// 후보가 아예 없으면 바로 트렌딩
if (candidates.isEmpty()) return getTrendingSpots(currentSpotIdx);

// Gemini 결과가 비면 트렌딩
if (aiResults.isEmpty()) return getTrendingSpots(currentSpotIdx);

// id 검증 통과 후 저장된 게 0건이어도 트렌딩
if (saved == 0) return getTrendingSpots(currentSpotIdx);
```

LLM 환각 방어도 폴백의 일부다. Gemini가 고른 `spot_idx`는 실제 후보 집합(`validIdx`)에 있는 것만 저장하고, 나머지는 버린다. 버려서 3건이 안 되면 `ensureExactRecommendationCount`가 후보·트렌딩으로 빈자리를 메워 항상 정확히 3건을 맞춘다.

### 4-2. 챗봇 절감·실패 폴백 (ChatbotService)

챗봇 한 번의 `ask()`는 본 LLM 호출 전에 두 번 빠져나갈 수 있다.

```text
요청
 ├─ fast-path 히트 (15자 이하 단순 네비) → LLM 없이 즉답
 ├─ 1차 분류 INAPPROPRIATE → 본 호출 생략, 안전 응답
 └─ 본 Gemini 호출
       ├─ 성공 → URL 화이트리스트 검증 후 반환
       ├─ 안전 필터로 본문 없음 → safetyBlockedResponse
       └─ 5xx·타임아웃·파싱 실패 → fallbackResponse (정적 링크 3개)
```

- **fast-path**: `ChatbotFastPathService.resolveOrNull`이 메시지를 정규화해 길이 15자 이하인 단순 요청만 키워드 매칭한다. 로그인/회원가입/홈/커뮤니티 등 정해진 라우트면 메시지·링크를 `chatbot_*.properties`에서 꺼내 즉답하고, 안 맞으면 null을 반환해 기존 LLM 파이프라인으로 넘긴다. 한국어뿐 아니라 login, community 같은 영어 키워드도 일부 매칭한다.
- **부적절 즉답**: `IntentContextService.classify`가 1차 분류에서 INAPPROPRIATE로 판정하면 본 Gemini 호출을 생략하고 `safetyBlockedResponse`(inappropriate=true)를 저장·반환한다. 욕설·잡담에 토큰을 쓰지 않는다.
- **LLM 실패 폴백**: 본 호출이 `HttpClientErrorException`(4xx/5xx)이나 일반 예외를 던지거나, 응답에 candidates가 없거나 파싱이 깨지면 `fallbackResponse`로 떨어진다. 이건 빈 답이 아니라 탐색·도우미·커뮤니티 정적 링크 3개와 추천 quickReply를 담은 의미 있는 응답이다.
- **분류 자체의 폴백**: 1차 분류용 Gemini 호출이 실패하면 `IntentContextService`는 불용어 사전 기반 규칙 토큰화로 키워드를 뽑는다. 분류 단계도 외부 호출에 묶이지 않는다.

:::warning 부적절 즉답은 보안 경계이기도 하다
부적절 메시지를 본 호출 전에 끊는 것은 비용 절감이자, 프롬프트 인젝션·잡담 유도가 본 LLM에 닿는 표면을 줄이는 방어다. 챗봇이 만든 링크 URL도 화이트리스트(`ALLOWED_URL_PATTERNS`)와 위험 스킴 차단(javascript:/data:/file:)을 통과해야만 사용자에게 노출된다.
:::

| 항목 | 상태 |
| --- | --- |
| 추천 캐시→Gemini→트렌딩 3단 폴백 | 구현됨 |
| Gemini가 고른 spot_idx 검증·정확히 3건 보장 | 구현됨 |
| 챗봇 fast-path(LLM 생략) | 구현됨 |
| 부적절 1차 분류 즉답 | 구현됨 |
| 챗봇 LLM 실패 시 정적 링크 폴백 | 구현됨 |
| 1차 분류 실패 시 규칙 기반 분류 폴백 | 구현됨 |
| 챗봇 URL 화이트리스트·위험 스킴 차단 | 구현됨 |
| 자동 재시도·지수 백오프 | 미구현 (단발 호출 후 폴백) |
| 모델 간 자동 페일오버(예: Gemini 실패 시 다른 모델로) | 미구현 (모델별 정적 폴백만) |
| AI 폴백 발생률·품질 정량 모니터링 | 미구현 (로그 기반, 향후 과제) |

:::warning 정직한 한계
현재 폴백은 "한 번 호출해 실패하면 안전 응답"이다. 재시도나 다른 제공자로의 페일오버는 없다. 트렌딩 폴백은 개인화가 아니므로 추천 품질이 떨어지고, 폴백이 얼마나 자주 발생하는지를 수치로 추적하는 체계도 아직 없다. 가용성은 확보했지만 "폴백 품질"을 측정·개선하는 단계가 다음 과제다.
:::

## 6. 면접 답변 3단계

1. **개념**: "외부 LLM은 통제 밖이라 느려지고 끊기고 가끔 깨진 JSON을 줍니다. 그래서 AI 기능마다 다단계 폴백을 두어, 실패해도 사용자에겐 빈 화면 대신 한 단계 낮은 유효 결과가 가게 했습니다."
2. **설계**: "추천은 캐시→Gemini→트렌딩 3단으로 하강하고 어느 단계든 채우면 반환합니다. 챗봇은 단순 네비는 fast-path로, 부적절 메시지는 1차 분류에서 끊어 본 호출을 아끼고, 본 호출이 실패하면 정적 링크 폴백으로 떨어집니다."
3. **트레이드오프**: "위 계층일수록 빠르고 싸고 개인화되지만, 아래일수록 일반적인 대신 항상 동작합니다. 지금은 단발 호출+폴백이라 재시도나 모델 페일오버, 폴백률 모니터링은 아직 없는 게 정직한 한계입니다."

## 7. 꼬리질문 + 모범답안

:::details Q1. 추천에서 트렌딩으로 떨어지는 분기가 여러 개인데, 왜 한 군데로 합치지 않았나요?
실패 원인이 단계마다 다르기 때문입니다. 후보 집합 자체가 비었을 때, Gemini가 빈 배열을 줬을 때, LLM이 고른 id가 검증을 다 통과 못 해 저장이 0건일 때는 서로 다른 지점에서 감지됩니다. 각 지점에서 즉시 같은 안전 응답(트렌딩)으로 수렴시키면, 불필요한 후속 처리를 건너뛰면서도 최종 결과는 항상 동일하게 유효합니다. 로그도 단계별로 남아 어디서 떨어졌는지 추적됩니다.
:::

:::details Q2. fast-path가 LLM을 건너뛰면 답변 품질이 떨어지지 않나요?
fast-path는 15자 이하의 명백한 단순 네비게이션만 대상으로 합니다. 로그인·커뮤니티·홈 같은 고정 라우트 안내는 LLM이 만들든 정적으로 만들든 답이 같으므로 품질 손실이 없습니다. 조금이라도 복합적이면 키워드 매칭이 실패해 null을 반환하고 기존 LLM 파이프라인으로 그대로 넘어갑니다. 즉 fast-path는 확실한 케이스만 가로채고 애매하면 안전하게 LLM에 위임합니다.
:::

:::details Q3. Gemini가 응답은 줬는데 본문이 없거나 깨진 JSON이면요?
두 경우를 분리해 처리합니다. 안전 필터로 차단돼 content가 없으면 inappropriate 안전 응답으로, candidates 누락이나 JSON 파싱 실패면 정적 링크를 담은 일반 폴백으로 떨어집니다. 추천 쪽은 한발 더 나가서 잘린 JSON에서도 중괄호 깊이를 세어 완성된 객체만 추려내는 복구 로직을 두어, 응답이 중간에 끊겨도 살릴 수 있는 항목은 살립니다.
:::

:::details Q4. 재시도나 다른 모델로의 페일오버는 왜 없나요?
의도적으로 단발 호출 + 폴백으로 단순화했습니다. 사용자가 기다리는 동기 경로라 재시도는 지연을 키우고, 모델 페일오버는 프롬프트·응답 스키마가 모델마다 달라 비용 대비 복잡도가 큽니다. 지금은 "한 번 시도, 실패하면 즉시 유효한 대체 결과"가 체감 품질에 더 유리하다고 판단했습니다. 다만 폴백률을 수치로 본 뒤, 특정 실패 유형이 잦으면 그 지점에만 짧은 재시도를 넣는 것이 자연스러운 다음 단계입니다.
:::

:::details Q5. 부적절 메시지를 본 호출 전에 끊는 게 비용 말고 다른 이점이 있나요?
보안 경계가 됩니다. 욕설·사이트 무관 잡담·프롬프트 인젝션 유도를 1차 분류에서 차단하면, 그런 입력이 본 LLM과 시스템 프롬프트에 닿는 표면이 줄어듭니다. 또 본 호출이 성공해 링크를 만들어도 화이트리스트와 위험 스킴 차단을 통과해야 노출되므로, LLM이 만든 위험한 URL이 사용자에게 가지 않습니다. 비용 절감과 안전이 같은 길목에서 동시에 처리됩니다.
:::

## 8. 직접 말해보기

다음을 소리 내어 설명해 보세요. 막히면 위 절로 돌아가 확인하세요.

- 추천의 3단 폴백을 단계·조건·비용으로 한 문장씩 설명하기
- 트렌딩 폴백이 발동되는 서로 다른 분기를 최소 3개 들기
- 챗봇이 본 LLM 호출을 건너뛰는 두 경로(fast-path, 부적절)를 구분해 말하기
