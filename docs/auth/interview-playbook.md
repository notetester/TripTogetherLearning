# 인증 도메인 면접 플레이북

> 인증·계정·보안 도메인을 1분/3분으로 압축해 말하고, "왜 세션인가 / 왜 위험 평가인가 / 왜 액션 토큰인가"를 근거와 함께 방어하고, 꼬리질문 10여 개를 미리 막는 한 장.

## 1. 한 줄 정의

이 페이지는 **인증 도메인을 면접에서 말로 풀어내기 위한 대본**이다. 새 기능 설명이 아니라, [개요](/auth/)부터 [WAF 어댑터](/auth/waf-adapters)까지 흩어진 내용을 면접 길이(1분/3분)와 질문 유형별로 재배열한 것이다.

## 2. 왜 이렇게 설계했나 (말하기 전에 잡아둘 큰 그림)

면접에서 인증 도메인은 "기능 나열"로 가면 진다. 채점자는 **선택의 이유**를 듣고 싶어 한다. TripTogether 인증은 세 가지 설계 결정으로 요약되고, 플레이북 전체가 이 셋을 변주한다.

| 결정 | 무엇을 골랐나 | 한 줄 이유 |
| --- | --- | --- |
| 인증 방식 | JWT가 아니라 **서버 세션**(`loginUser` = `UsersVO`) | JSP 서버 렌더링 + 단일 WAS, 즉시 무효화가 중요 |
| 비밀번호 | 암호화가 아니라 **BCrypt 단방향 해싱** | 복호화할 일이 없고, 솔트와 비용 인자로 무차별 대입을 늦춘다 |
| 안전성 | 인증 통과 후에도 **위험 평가 + 상태 머신**(ACTIVE/DORMANT/BLOCKED/DELETED) | 비밀번호가 맞아도 들여보내면 안 되는 경우가 있다 |

여기에 부가 결정이 두 개 더 붙는다. OAuth는 **state로 CSRF를 막고** 신규/기존 사용자를 분기하며, 메일 기반 흐름(가입 인증·아이디 찾기·비번 재설정)은 **하나의 일회성 액션 토큰**(`EMAIL_VERIFICATION`)으로 통일한다.

## 3. 1분 / 3분 대본

### 1분 버전 (엘리베이터)

> "TripTogether 인증은 **서버 세션 기반**입니다. 로그인에 성공하면 세션 속성 `loginUser`에 사용자 객체를 심고, 이후 인터셉터와 AOP가 그걸 읽어 로그인 여부와 권한을 판단합니다. 비밀번호는 평문 저장 없이 `BCryptPasswordEncoder`로 해싱하고, 로그인 시 `matches`로만 검증합니다. 소셜 로그인은 카카오·네이버·구글을 OAuth 2.0으로 붙였고, 인가 요청마다 `state`를 발급·검증해 CSRF를 막습니다. 그리고 비밀번호가 맞아도 끝이 아니라, 계정 상태가 ACTIVE인지 DORMANT·BLOCKED인지, 해당 IP가 로그인 잠금 상태인지까지 확인한 뒤에 세션을 발급합니다."

### 3분 버전 (구조 + 근거)

1분 버전을 말한 뒤, 아래 네 갈래로 살을 붙인다.

1. **로그인 한 번의 흐름** — `POST /auth/login`은 페이지 이동이 아니라 Ajax JSON 응답이다. `LoginRequestContext`(IP·User-Agent·requestId·flowTraceId)를 만들고, `AuthService.login`이 사용자 조회 → 위험 사전 판정 → 계정 상태/이메일 정책 검사 → BCrypt 비밀번호 검증 순으로 진행한다. 성공해야 `updateLastLoginAt` + 세션 저장이다.
2. **상태 머신** — `account_status`가 DELETED면 없는 사용자처럼, DORMANT면 휴면 해제 안내, BLOCKED면 사유·해제시각과 함께 차단 메시지를 돌려준다. 같은 비밀번호라도 응답이 갈린다.
3. **위험 평가의 골격** — 비밀번호 틀림은 계정 단위·IP 단위 카운터를 올리고, IP가 잠금 임계를 넘으면 다음 로그인은 사전 단계(`checkPreLogin`)에서 막힌다. 실제 AI/WAF 판정은 `LoginRiskAssessmentProvider`·`WafSyncProvider` 인터페이스로 열어 뒀고, 현재는 Mock 단계다(정직하게 구분).
4. **메일 기반 복구** — 아이디 찾기·비번 재설정·이메일 인증은 모두 `EMAIL_VERIFICATION`에 일회성 토큰을 발급하고 목적(purpose)으로 구분한다. 토큰에는 TTL이 있고, 검증 시 used로 마킹해 재사용을 막는다.

## 4. 동작 원리 (말로 쓰기 좋은 표·흐름)

### 로그인 의사결정 표 (이 표 하나면 분기 질문은 다 막힌다)

| 검사 단계 | 통과 못 하면 | 코드 위치 |
| --- | --- | --- |
| IP 로그인 잠금(사전) | 즉시 차단, 안내 메시지 | `LoginRiskPolicyService.checkPreLogin` |
| 사용자 존재 | USER_NOT_FOUND로 실패 기록 | `AuthServiceImpl.login` |
| account_status = DELETED | 없는 계정처럼 실패 | 동상 |
| 이메일 로그인 시 email_verified / email_login_enabled | 인증·허용 안 됐으면 실패 | `UsersVO` 플래그 |
| password_enabled | 로컬 비번 수단 없으면 실패 | 동상 |
| BCrypt matches | 틀리면 계정·IP 카운터 증가 | `handleWrongPassword` |
| account_status = BLOCKED / DORMANT | 차단·휴면 분기 응답 | 컨트롤러 분기 |

### OAuth 콜백 흐름 (한 호흡에)

```text
GET /auth/{provider}            → state 발급(세션 저장) + 인가 URL 리다이렉트
GET /auth/{provider}/callback   → consumeOauthState(state 검증·소비)
   → code로 access token 교환 → 프로필 조회 → SocialUserInfo로 정규화
   → 이미 연동된 계정? → UsersVO 반환(즉시 로그인)
   → 처음 보는 계정?   → SocialTempVO 반환 → 추가정보 입력(/auth/social/complete)
```

핵심 포인트 두 개: 카카오·네이버·구글의 응답 JSON 구조가 제각각이라 `SocialUserInfo`(provider·providerUserId·email·nickname)로 **흡수**한 뒤 공통 로직만 태운다는 것, 그리고 state는 한 번 쓰면 세션에서 제거(소비)해 재사용을 막는다는 것.

### 액션 토큰 한 종류로 세 흐름 (purpose로 분기)

```text
FIND_ID    → 아이디 힌트 확인 링크
RESET_PW   → 비밀번호 재설정 링크
PROFILE_EMAIL → 이메일 인증/변경 확인 링크
```

세 흐름 모두 `EMAIL_VERIFICATION`에 토큰을 넣고, TTL(분 단위, 런타임 설정에서 조정 가능)이 지나면 만료, 사용하면 used 처리. 새 토큰을 발급하기 전 같은 목적의 활성 토큰을 만료시켜 **유효 토큰 1개** 원칙을 지킨다.

## 5. 구현 상태 (면접에서 정직하게 선 긋기)

| 항목 | 상태 | 면접에서 이렇게 말한다 |
| --- | --- | --- |
| 세션 로그인/로그아웃/회원가입 | 구현됨 | "기본 인증은 전부 동작합니다" |
| 카카오·네이버·구글 OAuth(로그인·연동·해제) | 구현됨 | "state 검증과 소셜 토큰 폐기까지 처리합니다" |
| 이메일 인증·아이디 찾기·비번 재설정 | 구현됨 | "일회성 토큰 한 종류로 통일했습니다" |
| BCrypt 해싱 | 구현됨 | "평문 저장은 없습니다" |
| 휴면 전환·해제, 차단·차단 해제 신청 | 구현됨 | "상태 머신 + 배치 스케줄러로 관리합니다" |
| 위험 평가 골격(인터페이스·정책·카운터·검토 큐) | 구현됨 | "확장 지점은 다 깔려 있습니다" |
| 외부 AI 위험 판정 | 계획/Mock | "인터페이스는 있고 실제 모델은 미연동입니다" |
| WAF/CDN 동기화 | 부분/Mock | "Mock 어댑터가 기본이고 AWS·Cloudflare는 자격 증명이 필요합니다" |

:::tip 정직함이 점수다
"AI 위험 평가가 붙어 있나요?"에 "네"라고 답하면 다음 꼬리질문에서 무너진다. **골격은 구현, 외부 연동은 Mock**이라고 먼저 선을 그으면 오히려 설계 의도(확장에 열린 구조)를 설명할 기회가 된다.
:::

:::warning 공개 자료 보안 원칙
이 문서는 공개 저장소에 있으므로 실제 키·DB 호스트·내부 IP·계정은 담지 않는다. 설정값은 API_KEY, DB_HOST 같은 자리표시자로만 말한다. 모델명·기술명·클래스명은 공개해도 무방하다.
:::

## 6. 면접 답변 3단계 (질문 받으면 이 틀로)

어떤 인증 질문이든 **결정 → 이유 → 한계**의 3단계로 답하면 일관성이 생긴다.

1. **결정**: 무엇을 골랐는지 한 문장. (예: "세션 기반으로 갔습니다.")
2. **이유**: 프로젝트 제약과 묶어서. (예: "JSP 서버 렌더링에 단일 WAS라 세션이 단순하고 강제 무효화가 즉시 됩니다.")
3. **한계/대안**: 트레이드오프를 인정. (예: "수평 확장이 필요해지면 세션 스토어 외부화나 토큰 방식을 검토해야 합니다.")

## 7. 꼬리질문 + 모범답안

:::details Q1. 세션이면 서버를 여러 대로 늘릴 때 문제 없나요?
현재는 단일 WAS(embedded Tomcat) 전제라 인메모리 세션으로 충분합니다. 수평 확장이 필요해지면 세션을 외부 스토어로 빼거나(sticky session 또는 분산 세션) 토큰 방식으로 전환하는 선택지가 있고, 그건 인증 로직보다 인프라 결정입니다. 지금 구조에서 세션을 고른 이유는 강제 로그아웃·세션 무효화가 즉시 가능하다는 운영상 이점 때문입니다.
:::

:::details Q2. OAuth에서 state를 안 쓰면 무슨 일이 생기나요?
인가 콜백이 위조될 수 있습니다. 공격자가 자기 인가 코드로 피해자를 콜백시켜 계정을 바꿔치기하는 CSRF가 가능해집니다. 그래서 인가 요청마다 무작위 state를 발급해 세션에 저장하고, 콜백에서 받은 state가 저장본과 일치할 때만 진행한 뒤 그 state를 즉시 소비(제거)합니다. 일치하지 않으면 잘못된 접근으로 처리해 로그인 화면으로 되돌립니다.
:::

:::details Q3. BCrypt 대신 SHA-256 해시면 안 되나요?
SHA 계열은 빠른 게 장점인데, 비밀번호 저장에서는 빠른 게 단점입니다. 무차별 대입이 그만큼 빨라지니까요. BCrypt는 솔트를 내장하고 비용 인자(cost)로 의도적으로 느리게 만들어 대입 비용을 키웁니다. 같은 비밀번호라도 해시가 매번 달라지고, 우리는 복호화할 일이 없으니 단방향이면 충분합니다.
:::

:::details Q4. 비밀번호가 맞았는데도 로그인을 막는 경우가 있나요?
있습니다. account_status가 DELETED면 없는 계정처럼 막고, DORMANT면 휴면 해제 절차를 안내하고, BLOCKED면 사유·해제시각과 함께 차단 메시지를 줍니다. 이메일 로그인은 email_verified와 email_login_enabled가 모두 참이어야 하고, 해당 IP가 로그인 잠금 상태면 비밀번호 검증 이전 단계에서 막힙니다. 즉 비밀번호 일치는 여러 관문 중 하나일 뿐입니다.
:::

:::details Q5. 로그인 위험 평가에 AI가 실제로 붙어 있나요?
골격은 구현돼 있고 외부 AI 실연동은 아직 Mock입니다. 평가 로직은 `LoginRiskAssessmentProvider`라는 인터페이스 뒤에 두어, AI 모델이든 룰 엔진이든 상위 관제 연동이든 구현체를 Spring Bean으로 등록만 하면 정책 서비스가 자동으로 결과를 수집하도록 설계했습니다. 지금 실제로 도는 건 IP·계정 단위 실패 카운터 기반의 룰 잠금이고, 외부 모델 연동은 확장 지점만 열어둔 상태입니다.
:::

:::details Q6. 무차별 대입(brute force)은 어떻게 막나요?
비밀번호가 틀리면 계정 단위와 IP 단위로 실패를 누적합니다. 임계를 넘으면 해당 IP의 다음 로그인 시도를 사전 단계에서 잠그고, 사용자에게는 잠금 안내 메시지를 줍니다. 로그인에 성공하면 관련 카운터를 초기화합니다. 정책 값(임계·잠금 시간)은 `LOGIN_RISK_POLICY` 테이블과 런타임 설정으로 조정할 수 있어 코드 배포 없이 운영 중 튜닝이 됩니다.
:::

:::details Q7. 아이디 찾기에서 가입 여부가 노출되지 않나요?
응답 메시지를 의도적으로 동일하게 둡니다. 계정이 있든 없든 "일치하는 계정이 있으면 메일을 보냈다"는 같은 문구를 돌려줘 계정 열거(account enumeration)를 막습니다. 실제 발송 여부와 실패 사유는 사용자에게 노출하지 않고 보안 이력에만 남깁니다.
:::

:::details Q8. 비밀번호 재설정 링크를 가로채면 끝 아닌가요?
토큰은 일회성이고 TTL이 짧습니다. 한 번 사용하면 used로 마킹돼 재사용이 막히고, 새 토큰을 발급하면 같은 목적의 이전 활성 토큰을 만료시켜 유효 토큰을 하나로 유지합니다. 재설정이 완료되면 비밀번호는 다시 BCrypt로 해싱해 저장합니다. 링크 자체가 평문 비밀번호를 담지 않는다는 점도 핵심입니다.
:::

:::details Q9. 소셜 로그인만 한 사용자는 비밀번호가 없을 텐데요?
맞습니다. 소셜 최초 로그인은 사용자 객체 대신 임시 객체(`SocialTempVO`)를 반환해 추가 정보 입력 화면으로 보냅니다. 로컬 비밀번호 로그인은 password_enabled 플래그로 가능 여부를 따로 관리하고, 아이디·이메일·소셜 중 사용 가능한 로그인 수단이 최소 하나는 남도록 강제합니다. 마지막 로그인 수단을 지우려는 저장은 막습니다.
:::

:::details Q10. 차단된 계정은 영구히 못 들어오나요?
아닙니다. `SECURITY_ACTION_APPEAL` 기반 차단 해제 신청 흐름을 둡니다. 차단당한 사용자가 이메일 인증을 거쳐 이의를 제출하면 관리자가 검토해 수용·반려를 결정하고, 수용되면 차단 이력을 남기며 계정 상태를 복구합니다. 신청 자체에도 빈도 제한(쿨다운·일일 한도)을 둬 악용을 막습니다.
:::

:::details Q11. 로그아웃은 세션만 비우면 되는 거 아닌가요?
로컬 로그인은 세션 무효화로 끝나지만, 소셜 로그인은 외부 토큰까지 신경 씁니다. 네이버·구글은 보유한 access token을 폐기(revoke) 시도하고, 카카오는 로그아웃 URL로 보냅니다. 이 과정을 `flowTraceId`로 묶어 로그인/로그아웃 이력에 성공·실패 사유까지 남깁니다. 소셜 토큰을 안 지우면 외부 세션이 남아 다음 로그인이 자동으로 통과될 수 있어서입니다.
:::

:::details Q12. WAF 어댑터가 여러 개인데 왜 인터페이스로 추상화했나요?
위험 판정 결과를 외부 방화벽에 반영하는 방법이 환경마다 다르기 때문입니다. AWS WAF, Cloudflare, 게이트웨이 경유 등 대상이 달라도 우리 도메인은 "이 대상을 차단 동기화하라"는 동일한 명령만 알면 됩니다. 그래서 `WafSyncProvider` 인터페이스로 추상화하고, 차단 결정은 `WAF 동기화 큐`에 적재한 뒤 배치가 적합한 어댑터를 골라 처리합니다. 자격 증명이 없는 환경에서는 Mock 어댑터가 골격을 유지합니다.
:::

## 8. 직접 말해보기

아래를 소리 내어 답해 보고, 막히면 해당 챕터로 돌아간다.

1. (1분) 인증 도메인 전체를 60초로 설명해 보라. 세션·BCrypt·OAuth·상태 머신이 한 번씩 등장하는가?
2. 비밀번호가 맞아도 로그인이 막히는 경우를 세 가지 이상 들어 보라. → [로그인·세션](/auth/login-session)
3. OAuth에서 state가 없으면 생기는 공격을 이름과 함께 설명하라. → [OAuth 소셜 로그인](/auth/oauth-social)
4. 액션 토큰 하나로 세 흐름을 처리하는 구조를 그려 보라. → [이메일 인증·액션 토큰](/auth/email-verification-token)
5. "AI 위험 평가 붙어 있나요?"에 정직하게, 그러나 설계 의도가 드러나게 답하라. → [로그인 위험도 평가](/auth/login-risk-assessment)
6. 차단된 사용자가 합법적으로 돌아오는 경로를 끝까지 설명하라. → [차단 해제 신청](/auth/security-appeal)
