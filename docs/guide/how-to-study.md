# 왜 아는데 말이 막히나 — 학습/말하기 전략

> 머릿속에 든 지식과 면접관 앞에서 나오는 말은 다른 근육이다. 이 페이지는 TripTogether를 "이해"에서 "30초 안에 설명"으로 옮기는 훈련법과, 그 훈련을 위해 이 사이트를 쓰는 법을 정리한다.

## 1. 한 줄 정의

학습/말하기 전략이란, **코드를 읽어 이해한 내용을 면접에서 재현 가능한 문장으로 압축·인출하는 반복 훈련**이다. 핵심은 "다시 읽기"가 아니라 "닫고 말하기(인출)"다.

## 2. 왜 아는데 말이 막히나

면접에서 말이 막히는 건 지식이 없어서가 아니라, 지식이 **인식(보면 안다)** 수준에 머물러 **인출(닫고 말한다)** 수준까지 못 갔기 때문이다.

- **인식 ≠ 인출.** 코드를 다시 보면 "아 맞아" 하지만, 화면을 닫으면 첫 문장이 안 나온다. 인식은 단서가 있을 때의 기억, 인출은 단서 없이 꺼내는 기억이다. 면접은 항상 인출이다.
- **추상 용어를 구체 근거로 못 잇는다.** "AOP로 권한 처리했어요"까지는 말하는데, "그게 `AuthorizationAspect`가 `@RequireAdmin`을 보고 컷하는 거고, 인터셉터 체인 뒤에 도는 이유는…"으로 못 내려간다. 면접관은 한 단계 더 내려가는 순간을 본다.
- **전체 그림과 디테일을 동시에 못 든다.** TripTogether처럼 14~15개 모듈에 다중 AI 모델이 얽힌 프로젝트는, "이 프로젝트 뭐예요"에 큰 그림으로 답하고 "그 추천은 어떻게요"에 `SpotReview`·`Gemini 2.5 Flash`·3단 폴백으로 좁혀 답하는 **줌 인/줌 아웃**이 안 되면 한쪽만 말하게 된다.
- **남이 만든 모듈을 자기 말로 못 한다.** 4명이 도메인을 나눠 만든 팀 프로젝트라, 내가 직접 안 짠 영역(예: 다른 담당의 결제·신고)도 "프로젝트 전체"로 질문받는다. 읽기만 한 영역은 특히 인출이 약하다.

:::tip 핵심 전제
"한 번 더 읽으면 외워지겠지"는 거의 틀린다. 같은 시간이면 **읽기 1 : 말하기 3** 비율이 면접 점수를 더 올린다. 이 사이트의 페이지마다 끝에 퀴즈와 "직접 말해보기"가 붙어 있는 이유다.
:::

## 3. 무엇으로 훈련하나 — 이 사이트의 구조

이 사이트는 "읽고 끝"이 아니라 **인출을 강제하도록** 설계됐다.

| 구역 | 역할 | 인출 장치 |
| --- | --- | --- |
| [용어집](/glossary/) | API·DTO/VO·세션·AOP·소프트삭제 등 단어 단위 | 30초 정의 연습 |
| [백엔드](/backend/) | Spring Boot·MyBatis·JSP·인터셉터·AOP 등 기술 단위 | 기술→근거 클래스 매핑 |
| [도메인별 심화](/domains) | 인증·커뮤니티·문의·코스·탐색·AI·관리자·i18n | 페이지마다 "면접 답변 3단계" + 퀴즈 |
| [AI](/ai/) | 다중 모델·구조화 출력·폴백 전략 | 모델 선택 이유 말하기 |
| [전체 흐름](/flow/) | 아키텍처·사용자 여정·AI 통합·SSE | 줌 아웃 한 문장 |
| [담당별 보기](/by-area/) | 담당 라벨·도메인 태그 필터 | 내 범위만 묶어 복습 |

각 도메인 심화 페이지는 같은 골격을 쓴다: **한 줄 정의 → 왜 이렇게 설계 → 어떤 기술(실제 클래스·테이블) → 동작 원리 → 구현 상태(됨 vs Mock/계획) → 면접 답변 3단계 → 꼬리질문+모범답안 → 직접 말해보기 → 퀴즈.** 골격이 같으니, 한 도메인에서 익힌 답변 패턴을 다른 도메인에 그대로 옮길 수 있다.

:::tip 담당 태그로 내 범위만
프로젝트는 4명이 도메인을 나눠 만들었다. [담당별 보기](/by-area/)에서 담당 라벨(익명 A~D)과 도메인으로 필터링해 자기 범위를 묶어 보고, 그다음 [전체 흐름](/flow/)으로 남의 영역까지 넓힌다. 특정 사람을 "내 영역"이라 부르지 말고, 모든 도메인을 동등한 학습 대상으로 본다.
:::

## 4. 동작 원리 — 단어 정의에서 30초 답변까지

말하기는 **짧은 단위 → 긴 단위**로 쌓는다. 한 번에 "프로젝트 설명"을 외우려 하면 무너진다.

**4단계 사다리**

1. **단어 1문장 정의** — "MyBatis는 SQL을 XML 매퍼에 직접 쓰고 결과를 VO에 매핑하는 퍼시스턴스 프레임워크다." (용어집)
2. **기술 → 근거 1쌍** — "권한은 AOP로 한다. `AuthorizationAspect`가 `@RequireLogin`/`@RequireAdmin`을 보고 세션의 `loginUser`를 검사한다." (백엔드)
3. **기능 30초** — STAR 비슷한 4토막: 무엇을·왜·어떻게(실제 이름)·결과/한계.
4. **프로젝트 90초** — 도메인 4~5개를 한 문장씩 잇고, 한 도메인만 깊게 파고든 뒤 다시 빠져나온다.

**30초 답변 골격 (어디에나 적용)**

```text
[1] 무엇을  : "AI 여행 일정 자동 생성 기능을 만들었습니다."
[2] 왜      : "사용자가 빈 일정에서 막히지 않게, 한 번에 초안을 주려고요."
[3] 어떻게  : "GPT-4o-mini Structured Outputs로 JSON Schema(strict=true)를
              강제해 days[]→spots[] 구조를 받고, AiPlanResponseDTO로 파싱한 뒤
              TRAVEL_PLAN/PLAN_SPOT에 @Transactional로 저장합니다."
[4] 결과/한계: "스키마 강제라 파싱 실패가 거의 없습니다. 다만 응답 품질
              정량 평가 체계는 아직 없어 향후 과제입니다."
```

마지막 **한계 한 줄**이 신뢰를 만든다. 항공권은 Mock 프로바이더(`FlightOfferProvider` 인터페이스, 실제 외부 API 미연동), AI 품질 정량평가 부재, 모바일은 JSP 데스크톱 레이아웃 — 이런 정직한 경계를 미리 외워두면 꼬리질문에 흔들리지 않는다.

:::details 좋은 답변 vs 막히는 답변 (같은 질문)
**Q: "그 추천 기능 어떻게 만들었어요?"**

- ❌ 막힘: "음… AI 써서 추천해주는 거예요. Gemini였나…"
- ✅ 인출: "여행지 추천은 `Gemini 2.5 Flash`로 합니다. 사용자의 최근 30건 체류 로그와 태그 선호를 분석해 프롬프트를 만들고, **3단 폴백**으로 안정성을 확보했습니다 — DB 캐시 히트면 즉시 반환, 미스면 Gemini 호출, 그것도 실패하면 트렌딩 스팟으로 떨어집니다. 그래서 외부 API가 죽어도 화면이 빈 채로 안 남습니다."

차이는 지식량이 아니라, **근거 이름(모델·로그 범위·폴백 단계)을 닫고 꺼낼 수 있느냐**다.
:::

## 5. 8단계 학습 사이클

한 도메인 페이지를 다룰 때 도는 루프. 한 바퀴 30~40분.

| 단계 | 행동 | 인출 강도 |
| --- | --- | --- |
| 1. 줌아웃 | 페이지 "한 줄 정의"만 읽고 덮는다 | 낮음 |
| 2. 왜 묻기 | "왜 이렇게 설계?"를 먼저 스스로 답해본다 | 중간 |
| 3. 근거 확인 | 실제 클래스·테이블명(예: `SseEmitter`, `MYPAGE_FEED_NOTIFICATION`)을 본다 | 낮음 |
| 4. 흐름 그리기 | 동작 순서를 화살표로 손으로 그린다 | 높음 |
| 5. 경계 메모 | 됨 / Mock / 계획을 한 줄씩 적는다 | 중간 |
| 6. **닫고 말하기** | 페이지를 닫고 30초로 소리 내어 말한다 | **최고** |
| 7. 퀴즈 | 페이지 끝 QuizBox를 푼다 (틀리면 그 부분만 재인출) | 높음 |
| 8. 꼬리질문 | "면접관이라면 뭘 더 물을까"를 3개 만들어 답한다 | 높음 |

:::warning 가장 많이 건너뛰는 단계
6번(닫고 말하기)과 8번(꼬리질문 자작)이 점수를 만드는 핵심인데, 가장 자주 생략된다. 1~5번은 "공부한 느낌"을 주지만 인식까지만 올린다. 막힘은 6번에서만 풀린다.
:::

**스페이싱**: 같은 도메인을 하루에 몰아 3번 보지 말고, 1일·3일·7일 간격으로 짧게 다시 인출한다. 잊을 만할 때 꺼내는 게 가장 강하게 박힌다.

## 6. 면접 답변 3단계 (이 학습법 자체를 물어볼 때)

- **1단계(요약):** "이해와 설명은 다른 근육이라, 다시 읽기보다 닫고 말하는 인출 훈련 위주로 준비했습니다."
- **2단계(방법):** "도메인마다 한 줄 정의→근거 클래스→동작 흐름→구현 경계 순으로 정리하고, 매번 화면을 닫고 30초로 소리 내 말한 뒤 직접 만든 꼬리질문에 답하는 사이클을 돌렸습니다."
- **3단계(근거·한계):** "예컨대 AI 일정은 `GPT-4o-mini` Structured Outputs(strict), 추천은 `Gemini` 3단 폴백처럼 모델·근거를 이름으로 꺼낼 수 있게 외웠고, 항공권 Mock 같은 한계도 같이 외워 꼬리질문에 대비했습니다."

## 7. 꼬리질문 + 모범답안

:::details Q1. "공부는 했다는데, 직접 안 짠 모듈은 어떻게 설명하실 건가요?"
"팀이 도메인을 나눠 개발해서 제가 직접 구현하지 않은 영역도 있습니다. 그런 모듈은 코드를 읽고 **데이터 모델과 핵심 흐름 한 줄**까지는 인출할 수 있게 준비했습니다 — 예를 들어 신고는 `sourceType`/`sourceId`로 게시글·댓글·유저를 한 테이블에서 다루고 상태머신(검토중→처리완료/반려)으로 흐른다, 정도. 다만 제가 깊게 파고든 영역과 읽어서 아는 영역을 솔직히 구분해서 말합니다."
:::

:::details Q2. "암기한 답변과 진짜 이해를 어떻게 구분하죠?"
"꼬리질문에 한 단계 더 내려갈 수 있느냐로 구분된다고 봅니다. 그래서 저는 페이지마다 '면접관이라면 뭘 더 물을까'를 직접 만들어 답하는 연습을 했습니다. 예: AI 일정에서 '왜 Structured Outputs를 썼나'를 물으면, 자유 텍스트 파싱은 실패율이 높아 `strict=true` 스키마로 days/spots 구조를 강제해 파싱 안정성을 얻었다고 한 단계 내려가 답합니다."
:::

:::details Q3. "한정된 시간에 14~15개 모듈을 다 깊게 못 보면요?"
"우선순위를 둡니다. (1) 제가 주도한 도메인은 8단계 사이클로 깊게, (2) 나머지는 [전체 흐름](/flow/)과 [도메인 개요](/domains) 수준의 한 문장 인출까지만, (3) AI·인증·공통처럼 어느 면접에서나 단골인 영역은 추가로 깊게. 전부를 같은 깊이로 외우려다 다 얕아지는 걸 피합니다."
:::

:::details Q4. "면접에서 갑자기 머릿속이 하얘지면 어떻게 복구하나요?"
"30초 골격 4토막(무엇을·왜·어떻게·한계)을 앵커로 씁니다. 디테일이 안 떠올라도 '무엇을·왜'부터 말하기 시작하면 '어떻게'의 근거 이름이 따라옵니다. 처음 한 문장만 항상 같게 시작하도록 연습해두면 첫 막힘을 넘깁니다."
:::

:::details Q5. "이 학습 사이트는 누가 보라고 만든 건가요?"
"이 프로젝트를 면접에서 설명해야 하는 개발자가 대상입니다. 도메인별 심화로 깊이를, 담당 태그 필터로 자기 범위를 빠르게, 전체 흐름으로 모듈 연결을 잡게 구성했습니다. 모든 서술은 특정 개인이 아니라 도메인 단위로 중립적으로 쓰여 있어, 누구든 자기 학습 동선에 맞게 골라 볼 수 있습니다."
:::

## 8. 직접 말해보기

화면을 덮고, 소리 내어 답해보세요. (각 30초)

1. "인식과 인출의 차이가 뭐고, 면접에서 왜 인출이 중요한가?"
2. 30초 골격 4토막으로 **아무 도메인 하나**를 설명해보라 — 실제 클래스나 테이블 이름을 최소 하나 포함해서.
3. 방금 그 도메인에서 **구현됨 / Mock / 계획**을 한 줄씩 구분해 말해보라.
4. 자기가 직접 짜지 않은 도메인 하나를 골라, 데이터 모델과 흐름을 한 문장으로 인출해보라.

막히면 그 부분의 도메인 페이지로 가서 **6·7·8단계만** 다시 돌리세요. 처음부터 다시 읽지 마세요.
